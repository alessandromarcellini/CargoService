LABORATORIO DI INGEGNERIA DEI SISTEMI SOFTWARE

Introduction

Sprint 1 (available here) closed with the following results:
  • an executable prototype of the whole load cycle: request → answer policy (accepted / refused / retrylater with mutually exclusive branches) → 30 s engagement window → sustained detection → trip IOPort → slot5 (marking pause) → reserved slot → HOME, with the hold bookkeeping owned by cargoservice;
  • five automated unit TestPlans (TP1.1–1.5) plus the on-demand integration plan TP1.6 against the reused robotsmart26 navigation subsystem; the automation surfaced and fixed a real defect (the qak state-timer parsing bug that silently made the 30 s window eternal);

The goal of this sprint, as declared in the closing page of Sprint 1, is reported below.
The objective of Sprint 2 is to evolve the prototype towards the real devices and the deferred requirements:
  1. complete the Out of service logic with its sonar-driven cause: sustained D > DFREE for at least 3 s (the same persistence-counter mechanism already used for the detection), the 'Out of service' display message, and the safe stop of the robot; per the committente there is no recovery behaviour to implement;
  2. implement the LED blinking during the engagement window;
  3. move the sonar and LED logic towards the PicoW-like device (or its faithful simulation);
  4. realize the IOPort as a web GUI showing the hold state and the service messages;

Requirements

The requirements are unchanged with respect to Sprint 0.

The company asks us to build service named cargoservice that should work as follows.
The cargoservice is able to receive a request to load a container sent by some customer by using the pushbutton of the IOPort.

• It sends the answer retrylater, if the IOPort is currently occupied by a container or if the system is Out of service
• It rejects the request when the hold is already full, i.e. the slots1-4 are already occupied.
• Otherwise, it considers the system as engaged, detects a free slot and returns as answer the name of such a reserved slot. While engaged, the system must blink a Led.

When the request to load is accepted, the customer must move the container in the sensor area within prefixed amount of time (e.g. 30 secs), otherwise the systems becomes disengaged. Then, the cargoservice uses the cargorobot to move the container from the IOPort to the slot5 (for marking the container) and then to the reserved slot.

The service must also show on the display on the IOPort:
• the current state of the hold
• the message 'Service working', when it is all is going well
• the message 'Out of service' if the sonar sensor measures a distance D > DFREE for at least 3 secs (perhaps a failure of the sonar).

Requirement analysis

The requirement analysis of Sprint 0 (document), refined by the committente answers collected in Sprint 1 (document), remains valid and is not repeated. This sprint adds nothing to the glossary: every term it touches (sonar, LED, DFREE, Out of service, display, engaged) was already defined. What changes is which parts of the requirement text move from "deferred" to "in scope":
  • "While engaged, the system must blink a Led" — realized on the physical LED;
  • "the message 'Out of service' if the sonar sensor measures a distance D > DFREE for at least 3 secs" — realized with the physical sonar and logical wrapper (see Problem Analysis);
  • "the display on the IOPort" showing "the current state of the hold" and "Service working" — realized as a web GUI.

Additional clarifications for Sprint 2 (questions to the committente)

The move from simulated to physical devices raises points the requirement text does not fix. Following the method, they are recorded as questions; for each we state the interpretation adopted while the answer is pending.

  1. LED when the engagement dies. The requirement binds the blinking to "while engaged".
    Adopted interpretation: we read it strictly: the LED blinks from the acceptance until the system leaves the engaged condition for any reason — completion, timeout, or Out of service — and is off otherwise. To confirm.

  2. Web GUI concurrency. Sprint 1 fixed (with the committente) the single-connected-customer assumption. Does the web GUI change it — may several browsers be open on the IOPort page at once?
    Adopted interpretation: the assumption stands: the GUI represents the one physical IOPort of the hold, not a public web service. Several open pages would all mirror the same published port state; the service-side no-queueing guard remains the defense for requests that reach the busy service.

  3. Marker device. The Sprint 0 clarification (marker emulated by a fixed ~5 s pause in the movement logic) is assumed to still hold: this sprint does not introduce a physical marker.

  4. When is the sonar failure detectable? The Sprint 1 model has the service listen to the sonar only during the engagement window (proactive sonar, subscribed only once a request is taken in charge). Keeping that scoping in Sprint 2, the sonar-driven Out of service is detected only while a load request is pending (window open): a sonar that fails while the hold is disengaged or while the robot is travelling is not observed on the values.
    Adopted interpretation: we keep the Sprint 1 scoping — it is coherent with "the customer must place the container within 30 s" (the only phase in which the reading is functionally meaningful).

Problem analysis

This section extends the logical architecture of Sprint 1 to cover the sprint goal. Following the method, it deals only with abstract concepts (actors, messages, events, services, contexts); every binding to a concrete technology — the board, its firmware, the browser, the broker — is deferred to the Project section, where the physical/deployment architecture is designed and each choice is justified by a problem identified here.

Scope of the analysis

In scope: the sonar-driven Out of service detection (sustained D > DFREE for ~3 s) with its display evidence; the LED blinking during the engagement; the migration of sonar and LED onto a physically separate device node; the IOPort as a web GUI showing the current hold state and the service messages, which forces the push evolution of the state observation anticipated in Sprint 1.
Out of scope: any recovery from Out of service (none exists, per the committente); a physical marker device (emulated pause, per the Sprint 0 clarification); multi-customer access to the GUI (single-customer assumption confirmed above). Note on the "safe stop": the declared goal mentioned stopping the robot on a sonar failure; the analysis below (device-node section) shows this case cannot arise under the Sprint 1 listening scope, so no robot-halt mechanism is designed — the only Out of service reachable while the robot moves is the robot's own failure, already handled in Sprint 1.

The device node problem

The central novelty of this sprint is architectural, not functional: the sonar and the LED stop being model components simulated in a context and become physical devices on a separate computational node — a microcontroller-class board. This forces two questions that the Sprint 1 architecture never had to answer, treated as separate problems below.

How device measurements enter the system, and how commands reach the device

The board cannot host model components: no actor, no declared message, no FSM runtime lives there. Whatever the board emits is, from the model's viewpoint, raw data on a transport, not a declared event. Two positions are possible:
  • the device speaks the model's language — the firmware serializes the very message envelopes of the model, so its publications enter the actor world directly, with no intermediary;
  • the device stays domain-agnostic (chosen) — the firmware publishes plain measurements ("22.5") on its outbound channel and consumes plain commands ("start_blinking" / "stop_blinking") on its inbound one, knowing nothing of the model.
We choose the second, for three reasons. First, dependency direction: a firmware that emits model envelopes depends on the model grammar — every evolution of the message contract would require reflashing a physical board, the most expensive component to touch. Second, testability in isolation: a domain-agnostic device can be bench-tested with any generic transport client, with no knowledge of the application. Third, substitutability: the same raw contract serves any future device revision (or a simulator), while the application side absorbs every change.

Who interprets the measurements: the logic sonar

In Sprint 1 the interpretation of D (the persistence-counted detection) lived inside cargoservice, and the document explicitly promised to move it into a logic sonar, a model component executing on the cargoservice node. This sprint keeps that promise, because the sprint goal adds a second interpretation next to the detection: the sonar failure (D > DFREE sustained ~3 s) beside the presence (D < DFREE/2 sustained ~3 s). The logic sonar therefore:
  • intercepts the raw distance stream published by the device;
  • owns both persistence counters (presence and failure), symmetric to the Sprint 1 detection counter;
  • emits two declared, meaningful events: container_detected when the presence is sustained, out_of_service when the failure is sustained.
The listening scope, carried over from Sprint 1 and made explicit. Sprint 1 fixed that the sonar is heard only during the engagement window. We keep this, and it now governs both interpretations: the logic sonar evaluates the stream only while the window is open; outside the window the raw measurements are ignored (neither presence nor failure is computed). Concretely, cargoservice opens and closes the logic sonar's evaluation at the window boundaries (a start_detection / stop_detection pair). This does not contradict the committente's "proactive sonar, no trigger": the physical device keeps publishing on its own initiative, untriggered; what is scoped is the logic interpreter, an internal component, and the scoping is the honest cost of having moved the interpretation out of the service. The consequence — a sonar failure is detected only while a load is pending — is recorded as a committente question (Requirement analysis, question 4).

With this scope, cargoservice consumes container_detected and out_of_service only in its engaged/window states; no other state can even receive them, because the logic sonar is silent outside the window. The division of responsibility stays clean: the logic sonar decides what the measurements mean; cargoservice decides what the system does about it. It executes on the cargoservice node (as promised in Sprint 1): the interpretation serves exactly one consumer, the business logic, and co-location keeps the meaningful events on the shortest path while the raw stream stays confined to the transport.

The LED, deliberately, gets no logic actor. The symmetry is tempting and wrong. An intermediary component earns its existence with behaviour; the sonar side has behaviour (the two counters plus the scoped listening — a genuine interpretation FSM), the LED side has none: two commands, no state, no interpretation, no decision. A "logic led" would be a pure pass-through — precisely the adapter-as-overhead that Sprint 1's design argument rejected ("adapters would duplicate a capability the runtime already offers"). Therefore cargoservice publishes the blink commands directly on the device's inbound channel, exploiting the runtime's native support for the device transport. The rule that falls out, worth defending at the oral: adapt where there is interpretation; publish directly where there is only transport.

Out of service and a trip in progress: why no halt is designed

The declared goal spoke of stopping the robot when the sonar fails mid-trip. Under the listening scope just fixed, this situation cannot occur: the window closes the instant the container is detected and the trip begins, so the logic sonar is silent for the whole duration of the movement — no out_of_service can be emitted while the robot travels. The only way the system enters Out of service during a movement is therefore the robot's own failure (a leg answered targetUnreachable), which Sprint 1 already handles: abort_transfer discards the reservation and marks the hold terminally out of service. No revocation of a goal already given to the robot is needed, and no new interaction towards the robot is introduced — the Sprint 1 contract stays as small as it was. Both Out of service causes (sonar failure during the window, robot failure during the trip) still land in the same terminal state: the Sprint 1 unification argument continues to pay off.

The boundary problem: IOPort as a web GUI over two unidirectional channels

The second novelty: the IOPort stops being simulated and becomes a graphical interface running on a customer-facing node (a browser). The Sprint 1 ioport actor was, in substance, a console mock standing in for this GUI: it existed to exercise the load conversation before a real boundary existed. With the real boundary arriving, the mock is removed, and substituted by the real GUI.

A browser cannot be a model component: like the device node, the GUI is a domain-external node that exchanges data with the system over a transport layer. The boundary interaction is therefore redefined as two unidirectional channels, both natively supported by the model runtime:
  • inbound — load_requests: the GUI publishes the customer's load request (the pushbutton gesture) towards cargoservice;
  • outbound — hold_state: cargoservice publishes the observable state of the hold at every significant change — the engagement state, out-of-service evidence, and on an acceptance, the reserved slot name. This single publication is simultaneously the answer to the request and the display feed the requirement demands ("the current state of the hold", 'Service working', 'Out of service').
What this changes in the conversation model — stated openly. Sprint 0/1 fixed the load conversation as a request with three correlated replies. At the boundary, that request/reply correlation is now replaced by pub/sub: the "answer" is the hold_state publication that follows the decision. The Sprint 1 guarantee "exactly one reply per request" becomes "every decision on a request is followed by exactly one state publication that reflects it" — acceptance carries the reserved slot, refusal shows the full hold, retrylater shows the busy/out-of-service state. The three-way branch logic of handle_load_request is untouched; only the carrier of its outcome changes. The request/reply discipline survives only where its correlation semantics is actually needed: towards the robot (reachTarget).

The boundary carries declared messages in the model's msg(...) form — unlike the device topics. The board is a generic microcontroller that must stay ignorant of the model, so it speaks raw payloads. The GUI is the opposite kind of node: a boundary built for this system, whose whole job is the load conversation. It therefore speaks the model's grammar — what travels on load_requests and hold_state is a declared qak message envelope msg(...), not an ad-hoc string. This keeps the message-as-contract discipline unbroken at the very boundary the requirement is about: the request and the state/answer remain the same declared messages the model already speaks, inspectable and testable in the same terms as the rest of the system. The cost — the GUI builds and parses the envelope in JavaScript — is accepted; the concrete transport is a Design concern.

How the inhibition policy survives the removal of the ioport actor. The committente's answer ("inhibit the IOPort while an operation is in progress") was realized in Sprint 1 by the boundary actor answering retry later locally. With no boundary actor, the policy is realized on two levels, and the duplication is deliberate (defense in depth):
  • presentation level — the GUI disables the pushbutton whenever the last published hold_state is not disengaged, and shows the retry-later evidence. This is the presentation of an imposed state, not a decision: the page renders what the service published, exactly as the mock displayed what the service dispatched;
  • service level — the no-queueing guard of Sprint 1 (engaged_retrylater) becomes the primary defense, no longer a test-only hook: a request published while the service is engaged (a click racing the state publication, a stale page, a non-GUI client) is answered with the busy state on hold_state, never queued, never served late. Sprint 1 kept the guard "for the tests and as an extension hook"; this sprint is the future in which the hook fires in production.
The display forces the push evolution — and the tests follow it. Sprint 1 chose pull observation (get_hold) because the only consumer was a test asserting at chosen instants, and explicitly noted the choice would be revisited "when a consumer that wants continuous updates actually exists". That consumer now exists, and hold_state is the push: published by the single writer at every significant change (engagement transitions, slot occupation, out-of-service entry). Crucially, the tests adopt the same push channel: an automated plan subscribes to hold_state exactly as the GUI does, and asserts on the published states. This removes the pull request/reply pair (get_hold/hold_state-reply) that Sprint 1 had added purely for observability — dead boilerplate once a push feed exists. What remains of the test-support contract is only the controllability half, preset_hold, which sets the Given of a plan (and is the only way to reset the terminal out-of-service mark for the next plan — excluded from production, as the committente fixed no recovery). One state, written by one owner, read the same way by every observer; only the write for test setup is a separate, test-only interaction.

The engagement lifecycle with the new pieces. The GUI publishes a load request. cargoservice decides on the mutually exclusive branches (unchanged): on acceptance it reserves the slot, commands start_blinking, opens the logic sonar's listening (start_detection), publishes hold_state (engaged + reserved slot — the answer the customer sees) and opens the 30 s window. During the window the logic sonar may report either meaningful event: container_detected starts the trip; out_of_service discards the reservation and drops the hold into the terminal state. Window expiry (no detection) discards the reservation. Every one of these exits closes the listening (stop_detection), stops the LED, and publishes the new state. The trip then runs with the sonar silent; only a robot failure can send it to Out of service. The GUI shows 'Service working' or 'Out of service' as the published state dictates — the requirement's display sentence, realized.

Problems identified (each bound and justified in the Project section)

Four obstacles separate this logical architecture from a running system; each is addressed as a named section of the Project (Design):
  • the device transport — a carrier for raw device data between the board and the application, fitting a microcontroller-class node (lightweight, connection-loss tolerant, pub/sub so the device does not know its consumers);
  • the logic sonar and led commands technology;
  • the browser boundary — publishing on load_requests and observing hold_state reactively (no polling), given that browsers do not speak the transport natively; unlike the device topics, the boundary carries the model's declared msg(...) messages, so the load conversation stays inside the message-as-contract discipline;
  • the hardware-free simulation — a software device standing in for the board on the same transport, for deterministic demos and the automated TestPlans.

Test plans

Strategy. Same discipline as Sprint 1: automated plans with machine-checkable Given / When / Then and exact assertions. The observation channel changes: a plan now subscribes to the pushed hold_state (as the GUI does) instead of polling a pull reply, and sets its Given with preset_hold. Every device- and boundary-facing effect travels on an inspectable pub/sub transport, so a plan can both inject (publish raw distances or a boundary request as the device / GUI would) and observe (subscribe to the blink commands and to hold_state) with no test-only hook inside the components. The Sprint 1 suite is re-run as a regression gate, adapted only where the carrier changed (boundary request published on load_requests; assertions read from the hold_state feed).

TP2.1 — LED contract at the engagement boundaries

Given the hold empty (via preset_hold), a test subscriber on the LED command channel.
When a load request is accepted, then the engagement ends (either exit: detection + full trip, or window expiry).
Then exactly one start_blinking is observed at the acceptance and exactly one stop_blinking at the exit; no blink command is observed outside the engagement.

Given an accepted request (window open) on an empty hold.
When the test publishes raw distances with D > DFREE for at least 3 consecutive seconds, before any container detection.

Project (design)

Here the logical architecture is bound to concrete technologies and the physical / deployment architecture is designed: the transport, the physical device node and its firmware, the browser leg, and the hardware-free simulation. Each binding is justified by a problem of the analysis.

Physical architecture at a glance

Three domain-external nodes surround the model, all meeting it on one MQTT broker:
  • the PicoW device node (HC-SR04 sonar + LED), publishing raw distances and consuming raw blink commands;
  • the browser running the web GUI, publishing load requests and consuming the hold state;
  • the reused robot environment (virtual robot + robotsmart26), unchanged from Sprint 1.
The model nodes (ctxcargoservice hosting cargoservice + logic sonar + cargorobot; the external ctxrobotsmart) sit in the middle; the broker is the single meeting point of every heterogeneous node.

The device transport: MQTT (Mosquitto broker), raw device topics

The transport between the board and the application must be lightweight enough for a microcontroller, tolerate a device that appears/disappears, and preserve the proactive-publisher principle (the device must not know its consumers). This is the textbook profile of MQTT pub/sub through a broker; the broker is Eclipse Mosquitto, already present in the reused course stack (yamls/mosquitto.yml) — reuse justified: same protocol needs, zero new infrastructure. Alternatives considered: direct TCP/HTTP from the board (rejected: connection-oriented, couples the device to a consumer address, no pub/sub); CoAP observe (rejected for the device leg: the board firmware stack has first-class MQTT support and no equally mature CoAP one).

Two raw topics realize the domain-agnostic device contract, with names and payload syntax fixed by the firmware (pico_code/pico_mqtt.py):
  • sonar_data (device → system) — once per second the board publishes the measured distance as a plain ASCII decimal string, in centimetres, rounded to one decimal: firmware payload = str(round(distance, 1)) → e.g. "22.5". No unit suffix, no label, no envelope — the logic sonar parses the bare number (toDouble());
  • led_data (system → device) — the board subscribes and matches the payload as an exact ASCII command literal, one of "start_blinking" / "stop_blinking": firmware if msg == b"start_blinking" ... elif msg == b"stop_blinking". Any other payload is silently ignored, so cargoservice must publish exactly these two strings — they are the contract.
Note what is not on these topics: model envelopes. The firmware never sees the message grammar of the system (the dependency-direction argument).

The device node and its firmware

The physical device node is a Raspberry Pi Pico W carrying the HC-SR04 sonar and the LED; its firmware is pico_code/pico_mqtt.py (MicroPython). Its design:
  • connects to the Wi-Fi and to the broker (umqtt.simple, the de-facto standard lightweight client of the MicroPython ecosystem);
  • single non-blocking main loop: check_msg() polls the LED commands without blocking, the HC-SR04 is sampled via time_pulse_us every 1000 ms (the 1 Hz rate the persistence counters assume — the firmware constant and the model constant must stay aligned; recorded as a configuration invariant), the LED blinks on a 300 ms software timer while blinking is set;
  • a failed echo (timeout of time_pulse_us) publishes nothing: a silent sonar is indistinguishable from a slow one at the transport level, and the failure the requirement cares about (D > DFREE sustained) is decided by the logic sonar on the values, not on the gaps. A committente refinement (should silence itself mean failure) would be a logic-sonar change only — the firmware stays untouched, the domain-agnostic contract working as intended;
  • MicroPython over C/C++ SDK: the abstraction of the network stack removes accidental complexity, and the performance headroom of the board is irrelevant at 1 Hz.
The bench-test script pico_code/led_and_sonar_test.py (no network, console output) is the device's standalone verification — possible precisely because the firmware is domain-agnostic.

The logic sonar as a qak actor; LED commanded natively

The logic sonar is an ordinary QActor deployed in ctxcargoservice (the cargoservice node, per the Sprint 1 promise). It consumes the raw sonar_data topic through the qak runtime's MQTT support (the mqttBroker declaration binds the system to the broker), interprets the values with the two persistence counters while its listening is open, and emits the declared events container_detected / out_of_service on the ordinary event space; start_detection/stop_detection from cargoservice open and close it. The LED needs no actor: cargoservice publishes the raw payloads start_blinking / stop_blinking on led_data directly through the same native MQTT support — no interpretation to host, no component to justify.

The browser boundary: web GUI on MQTT-over-WebSocket, straight to the broker

The IOPort talks to cargoservice over MQTT-over-WebSocket, on two topics (load_requests inbound, hold_state outbound) whose payload is a declared qak message envelope msg(...). The motivations, in order of weight:
  • one transport for every external node. The board and the browser are both non-qak nodes that must meet the model; MQTT already carries the device leg, so reusing it for the boundary means one protocol, one broker, and no dedicated backend process to bridge the GUI — no HTTP or CoAP side-channel just for the browser;
  • pub/sub is exactly the display's need. The display must show the hold state continuously; a subscription to hold_state delivers every update by push, the instant the service publishes it — no polling, and the same feed the tests consume;
  • the envelope keeps the message-as-contract discipline (the key point). Because the payload is a declared msg(...), cargoservice receives on load_requests a genuine model message and publishes on hold_state a genuine model message — not an ad-hoc string. The load conversation, the part of the system the requirement is about, stays expressed in the same declared messages as everything else, and it is delivered by qak's native MQTT support (the mqttBroker declaration), the standard, well-understood path. The raw-topic ingestion the device leg needs does not apply here;
  • the WebSocket listener is the minimal enabler. Browsers cannot open raw MQTT/TCP sockets; Mosquitto exposes a second listener speaking MQTT-over-WebSocket (a few lines in the reused mosquitto.yml, typically port 9001) on the same broker and the same topics. A JavaScript MQTT client (e.g. mqtt.js from a CDN) connects to ws://host:9001, subscribes to hold_state and publishes on load_requests; the page is a static file, no server-side code.
Why the boundary differs from the device leg. The board speaks raw and the GUI speaks the envelope, and the asymmetry is principled: coupling a generic microcontroller to the model grammar is expensive and wrong (a reflash on every contract change), while the GUI exists only to serve this system's load conversation, so speaking its declared messages costs a little JavaScript and buys an unbroken message discipline at the boundary that matters most.

Alternatives considered. A dedicated HTTP/WebSocket backend bridging the browser to the actor world (as some teams do) — rejected: a new process whose only job the broker already does. CoAP observe directly from the browser — rejected: no native browser support. HTTP polling for the display — rejected: the degenerate case the push design exists to avoid.

Accepted trade-offs. (1) The GUI depends on the envelope grammar: it must build and parse msg(...) in JavaScript. Accepted, because it keeps one message discipline across the whole system and lets the boundary reuse qak's native delivery. (2) The page can publish while engaged (a stale or hostile client): the service-side no-queueing guard is the defense — the boundary is safe because the service is safe, not because the page behaves.

The hardware-free simulation: a simulated device in a separate model file

For deterministic demos and for the automated TestPlans the board must be replaceable by a software stand-in on the same transport, so that the whole chain downstream of the topics is byte-for-byte identical to a field run. This is realized as a separate qak model file (e.g. src/cargoservice_sprint2_simdevices.qak), deployed instead of the physical board, containing:
  • a physical-sonar stand-in actor that reads the values from the static table sonarDistances.txt (cyclic, editable — the deterministic source the committente asked for in Sprint 1) and publishes them on sonar_data, one per second, exactly as the board would;
  • a simulated LED actor that subscribes to led_data and renders the start_blinking/stop_blinking commands to the console.
Keeping the simulation in its own file (rather than a flag inside the main model) is the deliberate choice: the field model contains no simulation code, the simulation is swapped in purely at deployment by launching a different model, and the two never diverge because they meet only on the raw topics. The logic sonar, cargoservice and GUI are unaware of which side is publishing on sonar_data — the substitutability promised by the domain-agnostic device contract, made concrete.

Patterns (delta over Sprint 1)

  • Facade: cargoservice is now literally the single entry point — every domain-external node (GUI, device) reaches it through published channels it owns or consumes;
  • Adapter, only where behaviour exists: the logic sonar adapts the raw device stream to the declared contract and owns the interpretation FSM; the cargorobot keeps adapting the goal contract to the asset. The LED path deliberately has no adapter — the rule is stated in the analysis;
  • Observer: on three axes — raw device measurements, meaningful device events (container_detected/out_of_service), and the state push (hold_state);
  • Single writer, single read discipline: the hold state is written only by cargoservice and read by everyone the same way, over the push feed (GUI and tests alike); only the test-only preset_hold writes it, for controllability. Removing the pull channel deleted the second read discipline of Sprint 1 as dead boilerplate.

Why qak, re-checked. The Sprint 0/1 argument (executable metamodel: FSMs, declared contracts, location transparency) gains a new leg this sprint: the runtime's native MQTT support is what lets the model talk to two domain-external node kinds (board, browser) through the same broker with no bridge processes — the declared events travel intra-JVM in the unit tests and over MQTT towards the field, zero model changes. That is the metamodel earning its keep.

Deployment

Target arrangement for the Sprint 2 prototype. The field and simulated variants differ only in whether the physical board or the simulated-devices model publishes on sonar_data / consumes led_data:
  1. Clone the repository: github.com/alessandromarcellini/CargoService;
  2. Start the MQTT broker with both listeners — native MQTT for the board and the application, MQTT-over-WebSocket for the GUI: docker compose -f yamls/mosquitto.yml up -d from the repository root (the WebSocket listener is a configuration addition of this sprint);
  3. Start the robot environment and service: docker compose -f yamls/vrWithGui26.yaml up -d (virtual robot scene on http://localhost:8090), then ./gradlew run in robotsmart26;
  4. Start the application node from CargoService/Sprint2: ctxcargoservice (cargoservice + cargorobot + logic sonar); then open the GUI page in a browser (static file, connects to the broker's WebSocket listener);
  5. Devices — field variant: edit the header of pico_code/pico_mqtt.py (Wi-Fi SSID/password, broker IP = the LAN address of the machine running Mosquitto), flash it on the Pico W; the board joins the running system with no application-side change. Simulated variant: skip the board and launch the simulated-devices model (src/cargoservice_sprint2_simdevices.qak), which publishes the values of sonarDistances.txt on sonar_data and renders led_data to the console;
  6. Automated TestPlans: gradlew test from CargoService/Sprint2 (the test JVM needs the broker up for the device and boundary topics); integration variants (TP1.6, TP2.5-integration) on demand with the robot environment up.
Convenience — automatic scripts. The steps above can be run by hand, or with the two Windows PowerShell scripts in the repository root (mirroring the Sprint 1 ones): start_all2.ps1 brings the whole stack up in order (broker with both listeners → virtual robot → robotsmart26 → the Sprint 2 application → devices → the web GUI in the browser), and stop_all2.ps1 tears it all down. The devices variant is chosen with a flag: start_all2.ps1 alone runs the field variant (reminding you to flash the Pico W), while start_all2.ps1 -Simulated launches the simulated-devices model instead of the board.

Note. These steps are the deployment design; they become the verified end-user instructions when the sprint's implementation closes (the document is updated in a later version, as done in the previous sprints).
