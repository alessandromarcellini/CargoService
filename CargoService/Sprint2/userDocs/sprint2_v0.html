LABORATORIO DI INGEGNERIA DEI SISTEMI SOFTWARE

Introduction

Sprint 1 (available here) closed with the following results:
  • an executable prototype of the whole load cycle: request → answer policy (accepted / refused / retrylater with mutually exclusive branches) → 30 s engagement window → sustained detection → trip IOPort → slot5 (marking pause) → reserved slot → HOME, with the hold bookkeeping owned by cargoservice;
  • five committente answers folded into the model: robot failure → terminal Out of service (no recovery in production), HOME after every cycle, single-customer boundary with the no-queueing policy, and the sonar remodelled as a proactive, decoupled publisher of distance events;
  • five automated unit TestPlans (TP1.1–1.5) plus the on-demand integration plan TP1.6 against the reused robotsmart26 navigation subsystem; the automation surfaced and fixed a real defect (the qak state-timer parsing bug that silently made the 30 s window eternal);
  • one decision explicitly deferred with a promise: the interpretation of the sonar measurements would move from cargoservice into a dedicated logic sonar, "modeled as qactor on the cargoservice node, responsible of the sonar measurements evaluation" (Sprint 1, Behaviour section). This sprint keeps that promise.

The goal of this sprint, as declared in the closing page of Sprint 1, is reported below.
The objective of Sprint 2 is to evolve the prototype towards the real devices and the deferred requirements:
  1. complete the Out of service logic with its sonar-driven cause: sustained D > DFREE for at least 3 s (the same persistence-counter mechanism already used for the detection), the 'Out of service' display message, and the safe stop of the robot; per the committente there is no recovery behaviour to implement;
  2. implement the LED blinking during the engagement window;
  3. move the sonar and LED logic towards the PicoW-like device (or its faithful simulation);
  4. realize the IOPort as a web GUI showing the hold state and the service messages;
  5. verify on the field the policies fixed by the committente answers of this sprint (robot failure → terminal out of service, HOME when idle, ioport inhibition, proactive sonar), already part of the model and realized before this sprint's closure;

Hardware status note. The physical device work has already started ahead of this document: a Raspberry Pi Pico W hosts the HC-SR04 sonar and the LED, and a first firmware (pico_code/pico_mqtt.py) connects to the Wi-Fi, publishes one distance per second and reacts to blink commands. The Problem analysis below treats the board abstractly (a device node that cannot host model components); the concrete firmware is examined in the Project section, where its choices are bound to the problems identified here.

Requirements

The requirements are unchanged with respect to Sprint 0 and are reported verbatim.

The company asks us to build service named cargoservice that should work as follows.
The cargoservice is able to receive a request to load a container sent by some customer by using the pushbutton of the IOPort.

• It sends the answer retrylater, if the IOPort is currently occupied by a container or if the system is Out of service
• It rejects the request when the hold is already full, i.e. the slots1-4 are already occupied.
• Otherwise, it considers the system as engaged, detects a free slot and returns as answer the name of such a reserved slot. While engaged, the system must blink a Led.

When the request to load is accepted, the customer must move the container in the sensor area within prefixed amount of time (e.g. 30 secs), otherwise the systems becomes disengaged. Then, the cargoservice uses the cargorobot to move the container from the IOPort to the slot5 (for marking the container) and then to the reserved slot.

The service must also show on the display on the IOPort:
• the current state of the hold
• the message 'Service working', when it is all is going well
• the message 'Out of service' if the sonar sensor measures a distance D > DFREE for at least 3 secs (perhaps a failure of the sonar).

Requirement analysis

The requirement analysis of Sprint 0 (document), refined by the committente answers collected in Sprint 1 (document), remains valid and is not repeated. This sprint adds nothing to the glossary: every term it touches (sonar, LED, DFREE, Out of service, display, engaged) was already defined. What changes is which parts of the requirement text move from "deferred" to "in scope":
  • "While engaged, the system must blink a Led" — now realized, on the physical LED;
  • "the message 'Out of service' if the sonar sensor measures a distance D > DFREE for at least 3 secs" — now realized, as the second (sonar-driven) way of entering the terminal Out of service condition fixed by the committente answers of Sprint 1;
  • "the display on the IOPort" showing "the current state of the hold" and "Service working" — now realized as a web GUI.

Additional clarifications for Sprint 2 (questions to the committente)

The move from simulated to physical devices raises points the requirement text does not fix. Following the method, they are recorded as questions; for each we state the interpretation adopted while the answer is pending.

  1. Physical value of DFREE. The requirement uses DFREE symbolically. The prototype used an abstract value on the map scale (3); the physical HC-SR04 measures centimetres. What physical distance does DFREE denote at the real IOPort?
    Adopted interpretation: DFREE is a deployment configuration constant, not a business rule: the model reasons only in terms of the two thresholds D < DFREE/2 (presence) and D > DFREE (failure). Until the committente fixes the value, the field runs use DFREE = 30 cm (detection below 15 cm, failure above 30 cm), chosen from the bench geometry of our test rig.

  2. Out of service during the trip. The sonar failure can occur while the robot is moving a container. Must the robot be stopped immediately (the "safe stop" of the declared goal), or may it complete the current movement before the system freezes?
    Adopted interpretation: immediate stop. Since Out of service is terminal (committente answer of Sprint 1: no recovery, restoration is a maintenance intervention), there is no "resume" to design: the robot halts where it is, the reservation is discarded, and the hold is marked OUT_OF_SERVICE forever — only the test-support preset_hold may reset the mark, as pure controllability infrastructure of the TestPlans, excluded from production. A container possibly left in transit is a maintenance matter, coherently with the committente's framing.

  3. LED when the engagement dies. The requirement binds the blinking to "while engaged". We read it strictly: the LED blinks from the acceptance until the system leaves the engaged condition for any reason — completion, timeout, or Out of service — and is off otherwise. To confirm.

  4. Web GUI concurrency. Sprint 1 fixed (with the committente) the single-connected-customer assumption. Does the web GUI change it — may several browsers be open on the IOPort page at once?
    Adopted interpretation: the assumption stands: the GUI represents the one physical IOPort of the hold, not a public web service. Several open pages would all mirror the same published port state; the service-side no-queueing guard remains the defense for requests that reach the busy service.

  5. Marker device. The Sprint 0 clarification (marker emulated by a fixed ~5 s pause in the movement logic) is assumed to still hold: this sprint does not introduce a physical marker. To confirm.

Problem analysis

This section extends the logical architecture of Sprint 1 to cover the sprint goal. Following the method, it deals only with abstract concepts (actors, messages, events, services, contexts); every binding to a concrete technology — including the board, its firmware and the browser — is deferred to the Project section, where each choice is justified by a problem identified here.

Scope of the analysis

In scope: the sonar-driven Out of service detection (sustained D > DFREE for ~3 s) with its display evidence and the safe stop of the robot; the LED blinking during the engagement; the migration of sonar and LED onto a physically separate device node; the IOPort as a web GUI showing the current hold state and the service messages, which forces the push evolution of the state observation anticipated in Sprint 1.
Out of scope: any recovery from Out of service (none exists, per the committente), a physical marker device (emulated pause, per the Sprint 0 clarification), multi-customer access to the GUI (single-customer assumption confirmed above).

The device node problem

The central novelty of this sprint is architectural, not functional: the sonar and the LED stop being model components simulated in a context and become physical devices on a separate computational node — a microcontroller-class board. This forces three questions that the Sprint 1 architecture never had to answer.

(P2.1) How do device measurements enter the system, and how do commands reach the device? The board cannot host model components: no actor, no declared message, no FSM runtime lives there. Whatever the board emits is, from the model's viewpoint, raw data on a transport, not a declared event. Two positions are possible:
  • the device speaks the model's language — the firmware serializes the very message envelopes of the model, so its publications enter the actor world directly, with no intermediary;
  • the device stays domain-agnostic (chosen) — the firmware publishes plain measurements ("22.5") on its outbound channel and consumes plain commands ("start_blinking" / "stop_blinking") on its inbound one, knowing nothing of the model.
We choose the second, for three reasons. First, dependency direction: a firmware that emits model envelopes depends on the model grammar — every evolution of the message contract would require reflashing a physical board, the most expensive component to touch. Second, testability in isolation: a domain-agnostic device can be bench-tested with any generic transport client, with no knowledge of the application. Third, substitutability: the same raw contract serves any future device revision (or a simulator), while the application side absorbs every change.

(P2.2) Who interprets the measurements? The logic sonar — and why the LED gets no twin. In Sprint 1 the interpretation of D (the persistence-counted detection) lived inside cargoservice, and the document explicitly promised to move it into a logic sonar, a model component executing on the cargoservice node. This sprint keeps that promise, because the sprint goal doubles the interpretation work: alongside the detection (D < DFREE/2 sustained ~3 s, meaningful only during the engagement window) appears the failure (D > DFREE sustained ~3 s, meaningful always). Keeping both in cargoservice would force the business FSM to run a measurement-evaluation loop in every state of its lifecycle. The logic sonar therefore:
  • intercepts the raw distance stream published by the device;
  • owns both persistence counters (presence and failure), symmetric to the Sprint 1 detection counter;
  • emits two declared, meaningful events: container_detected when the presence is sustained, out_of_service when the failure is sustained.
cargoservice stops consuming distances entirely: it reacts to container_detected only inside the engagement window (events arriving in any other state are dropped by FSM semantics — the desired behaviour, unchanged from Sprint 1) and to out_of_service in every state, because the failure is state-independent. The division of responsibility is clean: the logic sonar decides what the measurements mean; cargoservice decides what the system does about it. It executes on the cargoservice node (as promised in Sprint 1): the interpretation serves exactly one consumer, the business logic, and co-location keeps the meaningful events on the shortest path while the raw stream stays confined to the transport.

The LED, deliberately, gets no logic actor. The symmetry is tempting and wrong. An intermediary component earns its existence with behaviour; the sonar side has behaviour (the two counters — a genuine interpretation FSM), the LED side has none: two commands, no state, no interpretation, no decision. A "logic led" would be a pure pass-through — precisely the adapter-as-overhead that Sprint 1's design argument rejected ("adapters would duplicate a capability the runtime already offers"). Therefore cargoservice publishes the blink commands directly on the device's inbound channel, exploiting the runtime's native support for the device transport. The rule that falls out, worth defending at the oral: adapt where there is interpretation; publish directly where there is only transport.

(P2.3) What does the sonar failure do to a trip in progress? The requirement's display message is the visible half of the condition; the invisible half is the robot. If the failure fires while the robot is moving a container, the service is inside a request/reply conversation with the cargorobot and, per the adopted interpretation (committente question 2), the movement must be interrupted, not completed. This is a genuinely new interaction problem: Sprint 1 never had to revoke a goal already given to the robot. Abstractly, the model needs a way to reach the robot executor outside the pending conversation — which is exactly what an out-of-band notification is for: the ordinary request/reply channel is occupied by the very conversation that must be aborted. The trip states of cargoservice therefore also listen for out_of_service, and on receiving it command the halt and take the abort exit (reservation discarded, hold marked OUT_OF_SERVICE — the same terminal landing already realized for the robot failure). One policy point deserves emphasis: both failure causes land in the same terminal state. This is the Sprint 1 unification argument paying off — no new condition, no new answer kind, no recovery machinery; the sonar cause is just a second entrance to a state the model already owns.

The boundary problem: IOPort as a web GUI over two unidirectional channels

The second novelty: the IOPort stops being simulated and becomes a graphical interface running on a customer-facing node (a browser). The Sprint 1 ioport actor was, in substance, a console mock standing in for this GUI: it existed to exercise the load conversation before a real boundary existed. With the real boundary arriving, the mock is removed, not evolved — and this must be stated as a revision of the Sprint 1 architecture, because Sprint 1 presented the ioport as the component realizing the committente's inhibition policy. Removing it forces that policy to be re-realized (below), not silently dropped.

A browser cannot be a model component any more than the board can: like the device node, the GUI is a domain-external node that exchanges data with the system over a transport. The boundary interaction is therefore redefined as two unidirectional channels, both natively supported by the model runtime:
  • inbound — load_requests: the GUI publishes the customer's load request (the pushbutton gesture) towards cargoservice;
  • outbound — hold_state: cargoservice publishes the observable state of the hold at every significant change — the engagement state, the working / out-of-service evidence, the slot map, and, on an acceptance, the reserved slot name. This single publication is simultaneously the answer to the request and the display feed the requirement demands ("the current state of the hold", 'Service working', 'Out of service').
What this changes in the conversation model — stated openly. Sprint 0/1 fixed the load conversation as a request with three correlated replies. At the boundary, that request/reply correlation is now replaced by pub/sub: the "answer" is the hold_state publication that follows the decision. The Sprint 1 guarantee "exactly one reply per request" becomes "every decision on a request is followed by exactly one state publication that reflects it" — acceptance carries the reserved slot, refusal shows the full hold, retrylater shows the busy/out-of-service state. The three-way branch logic of handle_load_request is untouched; only the carrier of its outcome changes. The request/reply discipline survives where its correlation semantics is actually needed: towards the robot (reachTarget) and in the test-support channel (get_hold/preset_hold).

How the inhibition policy survives the removal of the ioport actor. The committente's answer ("inhibit the IOPort while an operation is in progress") was realized in Sprint 1 by the boundary actor answering retry later locally. With no boundary actor, the policy is realized on two levels, and the duplication is deliberate (defense in depth):
  • presentation level — the GUI disables the pushbutton whenever the last published hold_state is not disengaged, and shows the retry-later evidence. This is the presentation of an imposed state, not a decision: the page renders what the service published, exactly as the mock displayed what the service dispatched;
  • service level — the no-queueing guard of Sprint 1 (engaged_retrylater) becomes the primary defense, no longer a test-only hook: a request published while the service is engaged (a click racing the state publication, a stale page, a non-GUI client) is answered with the busy state on hold_state, never queued, never served late. Sprint 1 kept the guard "for the tests and as an extension hook"; this sprint is the future in which the hook fires in production.
The display forces the push evolution. Sprint 1 chose pull observation (get_hold) because the only consumer was a test asserting at chosen instants, and explicitly noted the choice would be revisited "when a consumer that wants continuous updates actually exists". That consumer now exists, and hold_state is the push: published by the single writer at every significant change (engagement transitions, slot occupation, out-of-service entry). The pull channel (get_hold/preset_hold) remains untouched for the TestPlans: tests keep asserting at quiescent points with the synchronization-barrier semantics that made TP1.4 possible. Push and pull coexist because they serve different consumers with different needs — one state, two access disciplines.

Where does the business logic live? (re-asked, as the method requires)

Unchanged in its core: cargoservice keeps the answer policy, the engagement lifecycle, the trip sequencing and the hold bookkeeping. What this sprint moves out is interpretation and presentation, not business: the logic sonar owns the meaning of the measurements; the GUI owns nothing but rendering. Two additions to the service's responsibilities, both consequences of decisions above: it commands the LED at the engagement boundaries (the blinking is bound to its lifecycle, so it is the only component that knows when), and it publishes hold_state (single-writer principle: only the owner of the state may publish it).

Problems identified (each must be bound and justified in the Project section)

  1. P2.1 — a transport must carry raw device data between the board and the application, fitting a microcontroller-class node (lightweight, connection-loss tolerance, pub/sub to preserve the proactive-publisher principle);
  2. P2.2 — the logic sonar must consume that transport and live as an ordinary actor of the model; the LED command must reach the device with no intermediary, through the runtime's native transport support;
  3. P2.3 — the halt must actually interrupt the reused navigation executor mid-plan (to verify against the asset, not assume);
  4. P2.4 — a browser must publish on load_requests and observe hold_state reactively (no polling); browsers do not speak the device transport natively, and the payload format of the two boundary channels must be readable by both the page and the model runtime;
  5. P2.5 — the TestPlans must observe the new effects (blink commands, out-of-service entry, state publications) with the same automation discipline of Sprint 1.

Test plans

Strategy. Unchanged discipline: automated plans with machine-checkable Given / When / Then, exact assertions, observation at quiescent points via the pull channel (get_hold/preset_hold, kept precisely for this). New in this sprint: every device- and boundary-facing effect travels on an inspectable pub/sub transport, so a test can both inject (publish raw distances or a boundary request as the device / GUI would) and observe (subscribe to the blink commands and to hold_state) without any test-only hook entering the components — the P2.1 decision paying a testability dividend. The Sprint 1 suite (TP1.1–1.5) is re-run as a regression gate, adapted only where the boundary carrier changed.

TP2.1 — LED contract at the engagement boundaries

Given the hold empty, state disengaged, a test subscriber on the LED command channel.
When a load request is accepted, then the engagement ends (either exit: detection + full trip, or window expiry).
Then exactly one start_blinking is observed at the acceptance and exactly one stop_blinking at the exit; no blink command is observed outside the engagement.

TP2.2 — Sonar-driven Out of service at quiescence

Given the hold empty, state disengaged.
When the test publishes raw distances with D > DFREE for at least 3 consecutive seconds (as the failing device would).

TP2.3 — Failure debounce (no false Out of service)

Given the hold empty, state disengaged.
When the test publishes D > DFREE for less than 3 s (two readings), then normal values.
Then get_hold still shows disengaged / service working: a transient far reading must not kill the system. Symmetric to the Sprint 1 detection-persistence semantics.

TP2.4 — Detection through the logic sonar (regression of TP1.5 on the new chain)

Given an accepted request on an empty hold.
When the test publishes raw distances D < DFREE/2 for 3 consecutive seconds.

TP2.5 — Out of service during the trip halts the robot

Given an accepted request with the detection already fired (trip in progress, unit variant with the mock robot instrumented to report a pending goal).
When the test publishes a sustained failure stream mid-trip.
Then the robot receives the halt (mock observation), the pending leg closes with the failure reply, and get_hold shows out of service with no slot occupied (the reservation of the aborted transfer is discarded). Project (design)

Each problem of the analysis is now bound to a technology, and each binding is justified by that problem.

P2.1 → MQTT (Mosquitto broker), raw device topics

The transport between the board and the application must be lightweight enough for a microcontroller, tolerate a device that appears/disappears, and preserve the proactive-publisher principle (the device must not know its consumers). This is the textbook profile of MQTT pub/sub through a broker; the broker is Eclipse Mosquitto, already present in the reused course stack (yamls/mosquitto.yml) — reuse justified: same protocol needs, zero new infrastructure. Alternatives considered: direct TCP/HTTP from the board (rejected: connection-oriented, couples the device to a consumer address, no pub/sub); CoAP observe (rejected for the device leg: the board firmware stack has first-class MQTT support and no equally mature CoAP one).

Two raw topics realize the device contract of P2.1, with the names fixed by the firmware:
  • sonar_data — the board publishes one plain decimal distance (cm) per second;
  • led_data — the board subscribes; payloads start_blinking / stop_blinking.
Note what is not on these topics: model envelopes. The firmware never sees the message grammar of the system (the P2.1 dependency-direction argument).

The firmware (already written): what it does and why it is acceptable

pico_code/pico_mqtt.py (MicroPython on the Pico W) realizes the device node:
  • connects to the Wi-Fi and to the broker (umqtt.simple, the de-facto standard lightweight client of the MicroPython ecosystem);
  • single non-blocking main loop: check_msg() polls the LED commands without blocking, the HC-SR04 is sampled via time_pulse_us every 1000 ms (the 1 Hz rate the persistence counters assume — the firmware constant and the model constant must stay aligned; recorded as a configuration invariant), the LED blinks on a 300 ms software timer while blinking is set;
  • a failed echo (timeout of time_pulse_us) publishes nothing: a silent sonar is indistinguishable from a slow one at the transport level, and the failure the requirement cares about (D > DFREE sustained) is decided by the logic sonar on the values, not on the gaps. A committente refinement (should silence itself mean failure) would be a logic-sonar change only — the firmware stays untouched, which is P2.1 working as intended;
  • MicroPython over C/C++ SDK: the abstraction of the network stack removes accidental complexity, and the performance headroom of the board is irrelevant at 1 Hz.
The bench-test script pico_code/led_and_sonar_test.py (no network, console output) is the device's standalone verification — possible precisely because the firmware is domain-agnostic.

P2.2 → logic sonar as a qak actor; LED commanded natively

The logic sonar is an ordinary QActor deployed in ctxcargoservice (the cargoservice node, per the Sprint 1 promise). It consumes the raw sonar_data topic through the qak runtime's MQTT support (the mqttBroker declaration binds the system to the broker), interprets the values with the two persistence counters, and emits the declared events container_detected / out_of_service on the ordinary event space. The LED needs no actor: cargoservice publishes the raw payloads start_blinking / stop_blinking on led_data directly through the same native MQTT support — there is no interpretation to host, so there is no component to justify. The Sprint 1 table-driven simulated sonar is kept and re-pointed: it publishes on sonar_data like the board would, so the whole chain downstream of the topic is identical in simulated and physical runs — deterministic demos and TestPlans without the hardware, field runs without changing a line.

P2.3 → halting robotsmart26 mid-plan

The reused asset executes plans move-by-move in its internal executor. The halt is realized by the cargorobot adapter revoking the movement towards the asset; the concrete mechanism offered by robotsmart26 for interrupting the executor (its alarm-style event handling) must be verified against the asset before implementation — it is reused code, and the method forbids assuming reused behaviour. If the asset's interruption proves unreliable, the fallback is bounded degradation: the current leg completes (legs are short) and the abort fires at the leg boundary; the terminal outcome is identical, only the stop latency changes. Both variants keep the model contract (halt_robot, failure reply) unchanged — the uncertainty is confined to the adapter, which is exactly what the adapter is for.

P2.4 → web GUI on MQTT-over-WebSocket, straight to the broker

The boundary channels load_requests and hold_state are MQTT topics like the device ones — one transport for every domain-external node, one broker, no dedicated backend process. The browser leg has one obstacle: browsers do not open raw MQTT/TCP connections. The binding:
  • Mosquitto exposes a second listener, MQTT-over-WebSocket (a configuration addition to the reused mosquitto.yml); the page connects to it with a standard JavaScript MQTT client, subscribes to hold_state and publishes on load_requests;
  • reactivity is push by construction (the subscription delivers every state publication); no polling, no HTTP API, no server-side code for the GUI — the page is a static file;
  • alternatives considered: an HTTP/WebSocket backend bridging to the actor world (rejected: a new process whose only job the broker already does, since the boundary channels are plain topics); CoAP observe from the browser (rejected: no native browser support, would force the backend just rejected).
Accepted trade-offs, stated for the record. (1) The GUI depends on the payload format of the two topics: that format is part of the boundary contract and must be kept as stable as any declared message — the exact encoding (what the qak runtime natively consumes on an inbound topic, and the shape of the published state, e.g. a compact structured payload readable in JavaScript) is fixed at coding time after verifying the runtime's topic behaviour, and recorded then. (2) The page can technically publish while engaged (a stale or hostile client): the service-side guard is the defense — the boundary is safe because the service is safe, not because the page behaves.

Patterns (delta over Sprint 1)

  • Facade: cargoservice is now literally the single entry point — every domain-external node (GUI, device) reaches it through published channels it owns or consumes;
  • Adapter, only where behaviour exists: the logic sonar adapts the raw device stream to the declared contract and owns the interpretation FSM; the cargorobot keeps adapting the goal contract to the asset. The LED path deliberately has no adapter — the rule is stated in the analysis;
  • Observer: on three axes — raw device measurements, meaningful device events (container_detected/out_of_service), and the state push (hold_state);
  • Single writer, two read disciplines: the hold state is written only by cargoservice; read by push (GUI display) and by pull (tests) — each discipline justified by its consumer.

Why qak, re-checked. The Sprint 0/1 argument (executable metamodel: FSMs, declared contracts, location transparency) gains a new leg this sprint: the runtime's native MQTT support is what lets the model talk to two domain-external node kinds (board, browser) through the same broker with no bridge processes — the declared events travel intra-JVM in the unit tests and over MQTT towards the field, zero model changes. That is the metamodel earning its keep.

Deployment

Target arrangement for the Sprint 2 prototype (field variant with the physical board; the simulated variant differs only in step 5):
  1. Clone the repository: github.com/alessandromarcellini/CargoService;
  2. Start the MQTT broker with both listeners — native MQTT for the board and the application, MQTT-over-WebSocket for the GUI: docker compose -f yamls/mosquitto.yml up -d from the repository root (the WebSocket listener is a configuration addition of this sprint);
  3. Start the robot environment and service: docker compose -f yamls/vrWithGui26.yaml up -d (virtual robot scene on http://localhost:8090), then ./gradlew run in robotsmart26;
  4. Start the application from CargoService/Sprint2: the ctxcargoservice node (cargoservice + cargorobot + logic sonar) with its Gradle task; then open the GUI page in a browser (static file, connects to the broker's WebSocket listener);
  5. Physical devices: edit the header of pico_code/pico_mqtt.py (Wi-Fi SSID/password, broker IP = the LAN address of the machine running Mosquitto), flash it on the Pico W; the board joins the running system with no application-side change. Simulated variant: skip the board; the table-driven sonar (sonarDistances.txt) publishes on sonar_data instead;
  6. Automated TestPlans: gradlew test from CargoService/Sprint2 (unit variant, self-contained as in Sprint 1; the test JVM needs the broker up for the device/boundary channels); integration variants (TP1.6, TP2.5-integration) on demand with the robot environment up.
Note. These steps are the deployment design; they become the verified end-user instructions when the sprint's implementation closes (the document is updated in a later version, as done in the previous sprints).
