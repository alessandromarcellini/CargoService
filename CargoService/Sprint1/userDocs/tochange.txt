23. Resolutions applied to sprint1_v1.html (student decisions, 2026-07-20):
    (1) no reason-carrying inhibit needed: the retry-later reason travels in
        the load_retrylater(HOLDSTATE) argument (engaged|out_of_service);
        stated explicitly that the system NEVER queues load requests (the
        whole operation, window AND trip, answers immediate retrylater to
        requests that reach the service; ioport simply does not send while
        inhibited). Observation interactions (get_hold/preset_hold) remain
        deliberately served at quiescence.
    (2) HOME after EVERY cycle: with the ioport inhibited and no queueing,
        the committente condition "no pending request" is always true; the
        conditional skip-HOME branch could never trigger in production and
        would leave the robot parked at a slot.
    (3) out_of_service reset allowed FOR TESTS ONLY via preset_hold
        (controllability backdoor, P5); no recovery in production (Q4).
    (5) static distance table = deterministic PROTOTYPE/DEMO runs (and
        TP1.6); the unit TestPlans keep the silent mock + event injection.
    CODE IMPACT to remember at implementation: the no-queueing policy
    extends the retrylater guard to the TRIP states too -> each move state
    needs a reply-wait twin so that serving a request does not re-send the
    robot command.

24. Guard scope settled (student decision after requirement re-read):
    service-side retrylater guard stays on the ENGAGEMENT WINDOW only (the
    sole phase reachable by the asynchronous-inhibition race); during the
    TRIP the requirement branch "retrylater if the IOPort is currently
    occupied by a container" is realized by the INHIBITED IOPORT locally on
    the display (port occupied <=> operation in progress). No twin states on
    the robot-piloting states. New committente question added (port occupied
    with no operation in progress -> strict reading would need the service
    to remember the last sonar distance while disengaged; deferred).
