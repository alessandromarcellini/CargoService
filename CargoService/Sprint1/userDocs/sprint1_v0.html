Introduction

Sprint 0 (available here) closed with the following results:
  • the requirements of the committente were copied and every ambiguity was turned into an explicit question to the committente (with the answers recorded);
  • the requirement analysis classified every entity of the domain (POJO, enum, actor, service), fixed the message contract of the load conversation, and selected RobotSmart26 as the robot asset to reuse, with an explicit comparison of the four assets offered by the committente;
  • a first executable model of the system was produced;
  • a first automated Test Plan (TP0.1, load accepted on an empty hold) was implemented and executed; two further cases (TP0.2 refused, TP0.3 retrylater) were specified but explicitly deferred to Sprint 1, because they require the state of the hold to be observable and presettable from a test;
  • the Test Plan work revealed a defect in the model: the three answer branches of handle_load_request were written as sequential checks instead of mutually exclusive alternatives, so one request could produce two replies. Its correction was scheduled for this sprint.

The goal of this sprint, as declared in the closing page of Sprint 0, is reported below.
The goal of Sprint 1 is to create an executable prototype of the cargo service that implements the core business logic (loading and moving of the container), starting from the user load request up to the the container placement in the reserved slot, without worring about other small details like the Led blinking, the out of service possibility for the hold.

After finishing the sprint1 the system will be able to:
  1. Process load requests from the user, responding (with load_accepted, retry_later or load_refused) accordingly with the current state of the hold;
  2. Make the robot move across the hold to implement the container movement logic;
  3. Have a mock (simulation) of the devices of the ioport. The pushbutton will send messages to the system as the user would while the sonar will simulate the measuring of the distance when it's time to do so;
  4. The sonar outofservice logic and the blinking of the led are left for another sprint.

Scope note. Coherently with the declared goal, the Out of service logic and the LED blinking are outside the scope of this sprint and are deferred to Sprint 2. The messages related to the out-of-service branch are declared in the executable model as a skeleton for the next sprint: they are listed as such in this document, no component emits them in the flows of this sprint, and they are not covered by the Sprint 1 test plans.

Requirements

The requirements are unchanged with respect to Sprint 0 and are reported verbatim.

The company asks us to build service named cargoservice that should work as follows.
The cargoservice is able to receive a request to load a container sent by some customer by using the pushbutton of the IOPort.

• It sends the answer retrylater, if the IOPort is currently occupied by a container or if the system is Out of service
• It rejects the request when the hold is already full, i.e. the slots1-4 are already occupied.
• Otherwise, it considers the system as engaged, detects a free slot and returns as answer the name of such a reserved slot. While engaged, the system must blink a Led.

When the request to load is accepted, the customer must move the container in the sensor area within prefixed amount of time (e.g. 30 secs), otherwise the systems becomes disengaged. Then, the cargoservice uses the cargorobot to move the container from the IOPort to the slot5 (for marking the container) and then to the reserved slot.

The service must also show on the display on the IOPort:
• the current state of the hold
• the message 'Service working', when it is all is going well
• the message 'Out of service' if the sonar sensor measures a distance D > DFREE for at least 3 secs (perhaps a failure of the sonar).

Requirement analysis

The requirement analysis was carried out in depth during Sprint 0 (document) and its results are assumed valid for this sprint. In particular, the following results are reused as they are:
  • the nature of the system: distributed and heterogeneous, deployed on four logical nodes (business logic, ioport, robot, devices);
  • the entity model: IContainer/Container, ISlot/Slot, IHold/Hold, IDisplay/Display as passive objects; HoldState as an enumerative type; cargoservice as a service; ioport, sonar, led, cargorobot as actors;
  • the message contract of the load conversation (load_request with the three correlated answers) and the goal-level contract of the robot (reachTarget with its two correlated answers);
  • the committente clarifications already collected: the barcode of a container is 10 digits long; the dimensions of a container can be assumed integer; the functioning of the marker device can be emulated by a standard delay of 5 seconds in the container movement logic; the robot can be realized by reusing one of four assets provided by the committente (comparison in Sprint 0, choice: RobotSmart26).

Problem analysis

This section defines the logical architecture of the Sprint 1 slice of the system. Following the method, it deals only with abstract concepts (actors, messages, events, services, contexts): every binding to a concrete technology is deferred to the Project section, where each choice is justified by a problem identified here.

Scope of the analysis

In scope for this sprint:
  • the load conversation (request and the three answers) with mutually exclusive branches;
  • the engagement lifecycle, including the prefixed waiting window (30 s) after which the system becomes disengaged;
  • the detection of the container in the sensor area (simulated measurement);
  • the movement of the container: IOPort → slot5 (marking pause) → reserved slot → HOME, and the bookkeeping of the hold;
  • the observability of the hold state, needed to automate the test plans deferred from Sprint 0.
Out of scope, deferred to Sprint 2: the Out of service logic (sustained D > DFREE for 3 s and recovery), the LED blinking, the physical devices, the IOPort as a web GUI, and the strict 3 s persistence check of the detection measure.

Who knows the environment?

Before fixing the structure, one question must be answered explicitly, because it decides what the business model contains: who owns the knowledge of the physical environment? Two different kinds of spatial knowledge coexist in this system:
  • navigation knowledge: which cells are free or blocked by obstacles, and how to compute a route between two points. This knowledge is needed only to move;
  • business knowledge: which slots exist, whether each one is free or occupied, the state of the hold, and the coordinates of the relevant stations (IOPort, slot1..5, HOME). This knowledge is needed to answer requests and to command the trip.
Decision: the navigation knowledge (the map of the environment) belongs to RobotSmart26, which needs it to plan and execute movements; the business model owned by cargoservice keeps only the state of the slots, the state of the hold and the coordinates of the stations.
Why: single responsibility — the business logic never plans a route, so it has no reason to own a map; duplicating the map in two places would create a consistency problem between the business copy and the environment the robot actually navigates. The residual risk (the station coordinates kept by the business model must agree with the robot subsystem's environment) is a configuration-time alignment, accepted and logged.

Where does the business logic live?

The method requires this question to be answered explicitly. No pre-existing entity can host the business logic: the ioport is a boundary, the sonar is a detector, the cargorobot executes one goal at a time, the external robot subsystem only navigates. The business logic therefore lives in cargoservice, which owns:
  • the answer policy: accepted / refused / retrylater, decided against the hold model;
  • the engagement lifecycle: the transition disengaged → engaged on acceptance, and back on completion or timeout;
  • the trip plan: the sequence IOPort → slot5 (marking pause) → reserved slot → HOME, as the requirement sentence "the cargoservice uses the cargorobot to move the container..." dictates. Since the business model owns the station coordinates, cargoservice also resolves each named station into the position sent to the robot;
  • the hold state bookkeeping: reservation (kept in the service memory while engaged), occupation on deposit, discard of the reservation on timeout or failure.
The hold model is deliberately a passive object with a single owner (cargoservice): a single writer avoids consistency problems between concurrent requests, and a single place must be read to know the state of the system — which is exactly what the test plans need (P5).

Interaction

The interactions are formalized as abstract message exchanges. Three interaction types are used, each with a stated reason: a request/reply when the sender cannot proceed without the correlated outcome; a dispatch (fire-and-forget) when the sender only triggers an activity; an event (broadcast) when the producer must not know who consumes the information.

Declared but deferred. The event outOfService(none) and the corresponding out-of-service state of the service are declared in the executable model as a skeleton for Sprint 2; in this sprint no component emits the event.

The load conversation (abstract protocol). The ioport sends load_request. The cargoservice answers with exactly one of the three replies (P1). On acceptance the system becomes engaged and opens the waiting window; the detection event closes it and starts the trip; the window expiring closes it in the other direction, discarding the reservation. The trip is a sequence of reachTarget requests, each confirmed before the next: IOPort (pick up), slot5, a marking pause (~5 s, per committente clarification), the reserved slot, then HOME (committente question 2). After the deposit, the slot is marked occupied and the system returns disengaged.

Test plans

The source code of this sprint is available in the repository: CargoService/Sprint1 (executable model: cargoservice_sprint1.qak).

Strategy. As promised in Sprint 0, the test plans of this sprint are automated: an external test client plays the role of the ioport boundary, using the same request/reply contract, and the hold is made observable and presettable (P5) so that every case has a machine-checkable Given and Then. Every plan asserts the exact expected reply and that exactly one reply is produced (guard for P1). The observation interactions are served when the service is quiescent, so the state assertions are placed at the quiescent points of each scenario.

TP1.1 — Load accepted on an empty hold (inherited from TP0.1)

Given the system started, the hold empty, state disengaged.
When the test sends one load_request.
Then the answer is load_accepted(SLOTID) with SLOTID in 1..4, and exactly one reply is produced.

TP1.2 — Load refused when the hold is full (deferred from Sprint 0)

Given the hold preset via preset_hold(full) (slots1-4 occupied), state disengaged.
When the test sends one load_request.
Then the answer is load_refused, exactly one reply, and get_hold shows the four slots occupied and the state still disengaged.

TP1.3 — Retry later when the system is engaged (deferred from Sprint 0)

Given an accepted request (system engaged), at least one further slot free.
When the test sends a second load_request.
Then the answer is load_retrylater(engaged), exactly one reply, and the first conversation is unaffected.

TP1.4 — Engagement timeout discards the reservation

Given an accepted request (system engaged) with no container ever placed in the sensor area.
When the prefixed 30 s window expires (the test tolerates the wait).
Then get_hold shows the state disengaged and all slots still free: the reservation has been discarded exactly once (P8).

TP1.5 — Complete load cycle

Given an accepted request on an empty hold.
When the sonar detects the container (detection event emitted).
Then the trip IOPort → slot5 (marking pause) → reserved slot → HOME completes, and at the end get_hold shows the reserved slot occupied and the state disengaged. In the integration variant the trip runs against the reused robot subsystem; in the unit variant the cargorobot is replaced by a mock that confirms each target, so the assertion isolates the business logic.

Note. The out-of-service branch has no test plan in this sprint: it is out of the declared goal and is deferred to Sprint 2 together with its logic.

JUnit implementation (TestPlanSprint1)

The five plans are implemented as one JUnit 4 class, test/it/unibo/test/TestPlanSprint1.kt, runnable with a single command (gradlew test) and with no external process: the test JVM starts, once, the qak test configuration described by cargoservicetest.pl.

The test configuration (unit variant of TP1.5). The configuration deploys the real cargoservice together with two test doubles, all co-located in ctxcargoservice:
  • MockCargorobot replaces the robot adapter: it honours the same reachTarget/targetReached contract but confirms every target, so the assertions isolate the business logic (no robotsmart subsystem needed);
  • MockSonar absorbs checkMeasurement and never emits: the detection is injected by the test itself as an external containerPositioned event, which makes every plan deterministic (in particular TP1.4, which must observe no detection for 30 s — impossible with the random simulated sonar).
Because a qak actor is bound to its context only by the deployment description (location transparency), the service code under test is byte-for-byte the production one: only the .pl file changes. The test class plays the ioport boundary over TCP (port 8030) using the same request/reply contract of the model.

Changes introduced to make the TestPlans executable

  1. Mutually exclusive answer branches (P1). The defect inherited from Sprint 0 (sequential checks in handle_load_request, so one request could produce two replies) is corrected: the three branches are now an if / else-if / else chain, and every branch ends the conversation with exactly one reply.
  2. Observability/controllability messages (P5). New in the executable model: get_hold(none) with reply hold_state(HOLDSTATE) (e.g. disengaged_free_occupied_free_free) and preset_hold(full|empty) with reply preset_done(done), served by the states serve_get_hold/serve_preset_hold reachable from waiting only. They give every plan a machine-checkable Given and Then without touching the business conversation.
  3. Lowercase payload atoms (message-as-contract). The retrylater answer now carries load_retrylater(engaged) instead of load_retrylater(ENGAGED): an uppercase argument is parsed as a logical variable in the message term, so it is not a value a test (or the ioport) can match on. The ioport display logic was updated accordingly.
  4. Engagement window serves concurrent requests (Q3, TP1.3). A load_request received while engaged is answered load_retrylater(engaged) via the twin states engaged_retrylater/engaged_window (see the Behaviour table). Without this the FSM would defer the second request and answer it load_accepted at the end of the cycle, contradicting the adopted interpretation of question 3.
  5. Build changes. build.gradle: the test source set is mapped to the top-level test folder (the Gradle default src/test/kotlin would compile nothing and pass vacuously), JUnit 4 is declared, and the test task uses useJUnit() with the standard streams visible.
  6. Test configuration artifacts. New files: cargoservicetest.pl (deployment of the unit variant) and test/it/unibo/mock/MockCargorobot.kt, test/it/unibo/mock/MockSonar.kt.
  7. Timer bug found by TP1.4. The qak TimerActor resolves its owner actor from the timeout-event name with substringBeforeLast("_"): with a state named accept_load_request the generated event local_tout_cargoservice_accept_load_request is parsed as owner "cargoservice_accept_load", which does not exist — the timer thread loops forever and the 30 s window never expires. The timeout-event suffix was renamed to a single token (..._w30accept, ..._w30window); TP1.4 now measures the window at ~30.0 s. This is a concrete return on the "test plans must be automatable" requirement: the defect was invisible in any manual run.

How to run the TestPlans (step-by-step instructions for the committente)

The following procedure runs the automated TestPlans in their unit variant: it requires no Docker, no robot environment and no other process — the test JVM starts everything it needs by itself.

  1. Prerequisites. A Java JDK 21 installed and reachable (check with java -version); the project repository cloned from github.com/alessandromarcellini/CargoService. Nothing else is needed for the unit variant.
  2. Open a terminal in the sprint folder: CargoService/Sprint1 (the folder that contains build.gradle, cargoservicetest.pl and sysRules.pl). The tests must be launched from here: the test configuration reads the two .pl files from the working directory.
  3. Make sure port 8030 is free. If a cargoservice context (or a previous, interrupted test run) is still alive, close it first: the test JVM must be able to open TCP port 8030 itself. On Windows a leftover process can be found with netstat -ano | findstr 8030.
  4. Launch the tests:
    • Windows: gradlew.bat -b build.gradle test
    • Linux / macOS: ./gradlew -b build.gradle test
    The first launch may take a few minutes because Gradle downloads the declared dependencies; the following launches are fast.
  5. Wait for the run to finish — do not interrupt it. The whole suite lasts about one minute: TP1.4 intentionally waits the full 30 s engagement window (that wait is the test). The console shows the progress of each plan (passed/failed) together with the messages exchanged with the service.
  6. Read the outcome. Expected result: BUILD SUCCESSFUL with 5 tests completed, 0 failed. A browsable report is generated in build/reports/tests/test/index.html, with the standard output of every plan attached.

The test code

@FixMethodOrder(MethodSorters.NAME_ASCENDING)
class TestPlanSprint1 {

    companion object {
        private const val HOST = "localhost"
        private const val PORT = 8030          // ctxcargoservice, from cargoservice_sprint1.qak

        @BeforeClass
        @JvmStatic
        fun startTestConfiguration() {
            CommUtils.outcyan("TestPlanSprint1 | starting the qak test configuration (cargoservicetest.pl)")
            thread(isDaemon = true) {
                runBlocking {
                    QakContext.createContexts(
                        HOST, this, "cargoservicetest.pl", "sysRules.pl", "ctxcargoservice"
                    )
                }
            }
            Thread.sleep(5000)  // let the context server and the actors boot
        }
    }

    private var conn: Interaction? = null

    @Before
    fun setup() {
        try {
            conn = TcpClientSupport.connect(HOST, PORT, 10)
        } catch (e: Exception) {
            fail("TCP connection error towards the test configuration ($HOST:$PORT): ${e.message}")
        }
    }

    @After
    fun teardown() {
        conn?.close()
    }

    // ---------------------------------------------------------------- helpers

    private fun request(msgId: String, content: String): String {
        val envelope = "msg($msgId,request,testunit,cargoservice,$content,1)"
        val reply = conn!!.request(envelope)
        assertNotNull("no reply received for $msgId", reply)
        return reply.toString()
    }

    private fun loadRequest(): String = request("load_request", "load_request(none)")

    /** Presets the hold (Given); the reply is also a quiescence barrier. */
    private fun presetHold(config: String) {
        val reply = request("preset_hold", "preset_hold($config)")
        assertTrue("preset_hold($config) not confirmed (spurious reply pending?): $reply",
                   reply.contains("preset_done"))
    }

    /** Reads the observable hold description (Then + quiescence barrier). */
    private fun getHoldDescription(): String {
        val reply = request("get_hold", "get_hold(none)")
        assertTrue("expected hold_state but got (spurious reply pending?): $reply",
                   reply.contains("hold_state"))
        val m = Regex("""hold_state\(\s*([a-z_]+)\s*\)""").find(reply)
        assertNotNull("could not parse the hold description from: $reply", m)
        return m!!.groupValues[1]       // e.g. disengaged_free_occupied_free_free
    }

    /** Injects the sonar detection as the external event the model expects. */
    private fun injectContainerPositioned() {
        conn!!.forward("msg(containerPositioned,event,sonar,none,containerPositioned(none),1)")
    }

    private fun slotIdOf(reply: String): Int? =
        Regex("""load_accepted\(\s*(\d+)\s*\)""").find(reply)?.groupValues?.get(1)?.toIntOrNull()

    private fun occupiedCount(holdDesc: String): Int =
        Regex("occupied").findAll(holdDesc).count()

    // -------------------------------------------------------------- TestPlans

    /** TP1.1 — load accepted on an empty hold (inherited from TP0.1). */
    @Test(timeout = 60000)
    fun test1_loadAcceptedOnEmptyHold() {
        presetHold("empty")

        val reply = loadRequest()
        assertTrue("expected load_accepted but got: $reply", reply.contains("load_accepted"))
        assertFalse("wrong branch taken: $reply",
                    reply.contains("load_refused") || reply.contains("load_retrylater"))
        val slotId = slotIdOf(reply)
        assertNotNull("could not read the reserved slot id from: $reply", slotId)
        assertTrue("reserved slot id out of range (expected 1..4): $slotId", slotId!! in 1..4)

        // close the engagement (detection + full trip) so that the system is
        // quiescent again; getHoldDescription() is also the one-reply guard
        injectContainerPositioned()
        val hold = getHoldDescription()
        assertTrue("system not disengaged at the end of the cycle: $hold", hold.startsWith("disengaged"))
    }

    /** TP1.2 — load refused when the hold is full (deferred from Sprint 0). */
    @Test(timeout = 60000)
    fun test2_loadRefusedOnFullHold() {
        presetHold("full")

        val reply = loadRequest()
        assertTrue("expected load_refused but got: $reply", reply.contains("load_refused"))
        assertFalse("wrong branch taken: $reply",
                    reply.contains("load_accepted") || reply.contains("load_retrylater"))

        val hold = getHoldDescription()
        assertTrue("expected disengaged: $hold", hold.startsWith("disengaged"))
        assertEquals("the four slots must still be occupied: $hold", 4, occupiedCount(hold))
    }

    /** TP1.3 — retry later when the system is engaged (deferred from Sprint 0). */
    @Test(timeout = 90000)
    fun test3_retryLaterWhenEngaged() {
        presetHold("empty")

        val first = loadRequest()
        assertTrue("first request must be accepted: $first", first.contains("load_accepted"))
        Thread.sleep(500)   // let the service enter the engagement window

        val second = loadRequest()
        assertTrue("expected load_retrylater but got: $second", second.contains("load_retrylater"))
        assertTrue("the retrylater answer must carry the engaged state: $second",
                   second.contains("engaged"))
        assertFalse("wrong branch taken: $second",
                    second.contains("load_accepted") || second.contains("load_refused"))

        // the first conversation completes normally
        injectContainerPositioned()
        val hold = getHoldDescription()
        assertTrue("expected disengaged: $hold", hold.startsWith("disengaged"))
        assertEquals("only the slot of the FIRST conversation must be occupied: $hold",
                     1, occupiedCount(hold))
    }

    /** TP1.4 — engagement timeout discards the reservation. */
    @Test(timeout = 90000)
    fun test4_engagementTimeoutDiscardsReservation() {
        presetHold("empty")

        val reply = loadRequest()
        assertTrue("request must be accepted: $reply", reply.contains("load_accepted"))

        val start = System.currentTimeMillis()
        val hold = getHoldDescription()   // served only after the 30 s window
        val elapsed = System.currentTimeMillis() - start

        assertTrue("expected disengaged after the timeout: $hold", hold.startsWith("disengaged"))
        assertEquals("the reservation must be discarded (no slot occupied): $hold",
                     0, occupiedCount(hold))
        assertTrue("the window closed too early (${elapsed} ms)", elapsed >= 25000)
    }

    /** TP1.5 — complete load cycle (unit variant). */
    @Test(timeout = 90000)
    fun test5_completeLoadCycle() {
        presetHold("empty")

        val reply = loadRequest()
        assertTrue("request must be accepted: $reply", reply.contains("load_accepted"))
        val slotId = slotIdOf(reply)
        assertNotNull("could not read the reserved slot id from: $reply", slotId)

        injectContainerPositioned()
        val hold = getHoldDescription()   // deferred until the trip is over

        assertTrue("expected disengaged at the end of the cycle: $hold", hold.startsWith("disengaged"))
        val parts = hold.split("_")       // [disengaged, slot1, slot2, slot3, slot4]
        assertEquals("unexpected hold description: $hold", 5, parts.size)
        assertEquals("the RESERVED slot ($slotId) must be occupied: $hold",
                     "occupied", parts[slotId!!])
        assertEquals("exactly one slot must be occupied: $hold", 1, occupiedCount(hold))
    }

Project (design)

Only now each element of the logical architecture is bound to a concrete technology, and every binding is justified by a problem identified in the Problem analysis.

Why qak (not assumed, justified)

The system is a set of distributed state machines that interact only by messages. Plain Java/Kotlin offers no first-class construct for actors, messages, events or timed transitions: encoding them by hand means threads, sockets and callbacks that bury the model. qak is an executable metamodel: QActors are Moore FSMs, messages are declared contracts (Request/Reply/Dispatch/Event), timed transitions exist natively (whenTime), and the same QActor can be deployed as a nano-service in one JVM or as a distributed component without changing the model. This is exactly the gap between our Problem analysis and runnable code, which is why the executable model is written in qak (full argument in Sprint 0, "Why qak").

The hold model backtracking

In Sprint 1 we decided to remove the IHold interface and the separate Hold POJO, which contained the hold state and the mental model of the hold map. This decision was driven by the ready-to-use solution given by the committente, "RobotSmart26", that provides a sub-service inside for mantaining a map model of the environment where the robot operates. Coherently with the "Who knows the environment?" decision, the Hold is no longer a singular POJO kept inside the cargoservice, but becomes a business registry: state of the hold, state of the slots, and the coordinates of the stations. The hold map configuration is kept and updated owned by RobotSmart26. The alignment between the two descriptions of the environment is a configuration-time duty, verified during the integration run.

Patterns

  • Facade / delega: cargoservice is the single entry point of the system; ioport, tests and (in future) the web GUI all talk to it only. The devices are delegated nano-services, which decouples the boundary from the business logic.
  • Adapter (Port/Adapter): cargorobot adapts the domain contract reachTarget to RobotSmart26's moverobot; the business logic depends on the port, not on the asset (SOLID/Clean Architecture).
  • Dependency inversion on passive resources: the business logic depends on IDisplay, ISlot and IContainer, not on the concrete classes.
  • Observer (events): the detection is a broadcast event, so the sonar does not know its consumers.

Deployment

Steps for an end user to run the Sprint 1 prototype:
  1. Clone the project repository: github.com/alessandromarcellini/CargoService;
  2. Start the robot environment (reused committente assets): in robotsmart26/yamls run docker compose -f unibobasic26.yaml up, then open http://localhost:8090/ in a browser to see the virtual robot scene;
  3. Start the robot service: in the robotsmart26 directory run ./gradlew run;
  4. Start the prototype contexts from CargoService/Sprint1, one process per context, with gradlew -b build.gradle run, setting application.mainClassName to the context main: it.unibo.ctxcargoservice.MainCtxcargoserviceKt, it.unibo.ctxdevices.MainCtxdevicesKt, it.unibo.ctxrobot.MainCtxrobotKt, and last it.unibo.ctxioport.MainCtxioportKt (the ioport is the manual demo driver);
  5. To run the automated Test Plans (unit variant): gradlew -b build.gradle test from CargoService/Sprint1. Nothing else has to be running: the test JVM starts by itself the test configuration described by cargoservicetest.pl (the real cargoservice plus the two mocks).
