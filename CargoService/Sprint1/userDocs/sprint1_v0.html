Introduction

Sprint 0 (available here) closed with the following results:
  • the requirements of the committente were copied verbatim and every ambiguity was turned into an explicit question to the committente (with the answers recorded);
  • the requirement analysis classified every entity of the domain (POJO, enum, actor, service), fixed the message contract of the load conversation, and selected RobotSmart26 as the robot asset to reuse, with an explicit comparison of the four assets offered by the committente;
  • a first executable model of the system was produced;
  • a first automated Test Plan (TP0.1, load accepted on an empty hold) was implemented and executed; two further cases (TP0.2 refused, TP0.3 retrylater) were specified but explicitly deferred to Sprint 1, because they require the state of the hold to be observable and presettable from a test;

The goal of this sprint, as declared in the closing page of Sprint 0, is reported verbatim below.
The goal of Sprint 1 is to create an executable prototype of the cargo service that implements the core business logic (loading and moving of the container), starting from the user load request up to the the container placement in the reserved slot, without worring about other small details like the Led blinking, the out of service possibility for the hold or actually placing physical stuff on top of a PICO W (simulating these elements).

After the competion of sprint1 the system will be able to:
  1. Process load requests from the user, responding (with load_accepted, retry_later or load_refused) accordingly with the current state of the hold;
  2. Make the robot move across the hold to implement the container movement logic;
  3. Have a mock (simulation) of the devices of the ioport. The pushbutton will send messages to the system as the user would while the sonar will simulate the measuring of the distance when it's time to do so;
  4. The sonar outofservice logic and the blinking of the led are left for another sprint.

Scope note. Coherently with the declared goal, the Out of service logic and the LED blinking are outside the scope of this sprint and are deferred to Sprint 2. The messages related to the out-of-service branch are declared in the executable model as a skeleton for the next sprint: they are listed as such in this document, no component emits them in the flows of this sprint, and they are not covered by the Sprint 1 test plans.

Requirements

The requirements are unchanged with respect to Sprint 0 and are reported verbatim.

The company asks us to build service named cargoservice that should work as follows.
The cargoservice is able to receive a request to load a container sent by some customer by using the pushbutton of the IOPort.

• It sends the answer retrylater, if the IOPort is currently occupied by a container or if the system is Out of service
• It rejects the request when the hold is already full, i.e. the slots1-4 are already occupied.
• Otherwise, it considers the system as engaged, detects a free slot and returns as answer the name of such a reserved slot. While engaged, the system must blink a Led.

When the request to load is accepted, the customer must move the container in the sensor area within prefixed amount of time (e.g. 30 secs), otherwise the systems becomes disengaged. Then, the cargoservice uses the cargorobot to move the container from the IOPort to the slot5 (for marking the container) and then to the reserved slot.

The service must also show on the display on the IOPort:
• the current state of the hold
• the message 'Service working', when it is all is going well
• the message 'Out of service' if the sonar sensor measures a distance D > DFREE for at least 3 secs (perhaps a failure of the sonar).

Requirement analysis

The requirement analysis was carried out in depth during Sprint 0 (document) and its results are assumed valid for this sprint. In particular, the following results are reused as they are:
  • the nature of the system: distributed and heterogeneous, deployed on four logical nodes (business logic, ioport, robot, devices);
  • the entity model: IContainer/Container, ISlot/Slot, IHold/Hold, IDisplay/Display as passive objects; HoldState as an enumerative type; cargoservice as a service; ioport, sonar, led, cargorobot as actors;
  • the message contract of the load conversation (load_request with the three correlated answers) and the goal-level contract of the robot (reachTarget with its two correlated answers);
  • the committente clarifications already collected: the barcode of a container is 10 digits long; the dimensions of a container can be assumed integer; the functioning of the marker device can be emulated by a standard delay of 5 seconds in the container movement logic; the robot can be realized by reusing one of four assets provided by the committente (comparison in Sprint 0, choice: RobotSmart26).

Problem analysis

This section defines the logical architecture of the Sprint 1 slice of the system. Following the method, it deals only with abstract concepts (actors, messages, events, services, contexts): every binding to a concrete technology is deferred to the Project section, where each choice is justified by a problem identified here.

Scope of the analysis

In scope for this sprint:
  • the load conversation (request and the three answers) with mutually exclusive branches;
  • the engagement lifecycle, including the prefixed waiting window (30 s) after which the system becomes disengaged;
  • the detection of the container in the sensor area (simulated measurement);
  • the movement of the container: IOPort → slot5 (marking pause) → reserved slot → HOME, and the bookkeeping of the hold;
  • the observability of the hold state, needed to automate the test plans deferred from Sprint 0.
Out of scope, deferred to Sprint 2: the Out of service logic (sustained D > DFREE for 3 s and recovery), the LED blinking, the physical devices, the IOPort as a web GUI, and the strict 3 s persistence check of the detection measure.

Who knows the environment?

Before fixing the structure, one question must be answered explicitly, because it decides what the business model contains: who owns the knowledge of the physical environment? Two different kinds of spatial knowledge coexist in this system:
  • navigation knowledge: which cells are free or blocked by obstacles, and how to compute a route between two points. This knowledge is needed only to move;
  • business knowledge: which slots exist, whether each one is free or occupied, the state of the hold, and the coordinates of the relevant stations (IOPort, slot1..5, HOME). This knowledge is needed to answer requests and to command the trip.
Decision: the navigation knowledge (the map of the environment) belongs to the robot subsystem, which needs it to plan and execute movements; the business model owned by cargoservice keeps only the state of the slots, the state of the hold and the coordinates of the stations.
Why: single responsibility — the business logic never plans a route, so it has no reason to own a map; duplicating the map in two places would create a consistency problem between the business copy and the environment the robot actually navigates. The residual risk (the station coordinates kept by the business model must agree with the robot subsystem's environment) is a configuration-time alignment, accepted and logged.

Where does the business logic live?

The method requires this question to be answered explicitly. No pre-existing entity can host the business logic: the ioport is a boundary, the sonar is a detector, the cargorobot executes one goal at a time, the external robot subsystem only navigates. The business logic therefore lives in cargoservice, which owns:
  • the answer policy: accepted / refused / retrylater, decided against the hold model;
  • the engagement lifecycle: the transition disengaged → engaged on acceptance, and back on completion or timeout;
  • the trip plan: the sequence IOPort → slot5 (marking pause) → reserved slot → HOME, as the requirement sentence "the cargoservice uses the cargorobot to move the container..." dictates. Since the business model owns the station coordinates, cargoservice also resolves each named station into the position sent to the robot;
  • the hold bookkeeping: reservation (kept in the service memory while engaged), occupation on deposit, discard of the reservation on timeout or failure.
The hold model is deliberately a passive object with a single owner (cargoservice): a single writer avoids consistency problems between concurrent requests, and a single place must be read to know the state of the system — which is exactly what the test plans need (P5).

Interaction

The interactions are formalized as abstract message exchanges. Three interaction types are used, each with a stated reason: a request/reply when the sender cannot proceed without the correlated outcome; a dispatch (fire-and-forget) when the sender only triggers an activity; an event (broadcast) when the producer must not know who consumes the information.

The load conversation (abstract protocol). The ioport sends load_request. The cargoservice answers with exactly one of the three replies (P1). On acceptance the system becomes engaged and opens the waiting window; the detection event closes it and starts the trip; the window expiring closes it in the other direction, discarding the reservation. The trip is a sequence of reachTarget requests, each confirmed before the next: IOPort (pick up), slot5, a marking pause (~5 s, per committente clarification), the reserved slot, then HOME (committente question 2). After the deposit, the slot is marked occupied and the system returns disengaged.

Problems and obstacles

The problems that separate this logical architecture from a working prototype. Each of them is bound to a solution in the Project section.
  1. P1 — Mutually exclusive answers. The Sprint 0 model can produce two replies to one request (sequential checks). The answer policy must be restructured as mutually exclusive alternatives; the test plans guard this with "exactly one reply" assertions.
  2. P2 — Timed engagement window. The 30 s window must be modelled as a timed transition of the state machine, not as a busy wait: an actor stuck in a loop stops handling messages.
  3. P3 — Detection triggering. Someone must initiate the measurement. Chosen: the cargoservice triggers the sonar upon engagement (dispatch), the sonar measures repeatedly and answers with a broadcast event. The 3 s persistence check of the measure is consciously simplified in this sprint (deferred with the failure logic).
  4. P4 — Split of the environment knowledge. The navigation map belongs to the robot subsystem; the business model keeps only slot states, hold state and station coordinates ("Who knows the environment?"). Consequently the trip and the coordinate resolution are business logic, and the robot receives one goal position at a time. The residual risk is the configuration-time alignment between the station coordinates and the robot's environment.
  5. P5 — Observability and controllability. TP0.2/TP0.3 were deferred exactly because the hold state could not be read or preset from outside. The hold model must become an observable resource, and its occupancy presettable, for the tests to establish their Given and assert their Then.
  6. P6 — Marking without a marker component. The committente stated that the marker can be emulated by a ~5 s pause in the movement logic: no marker component is introduced in this sprint (a deliberate difference from other teams, justified by our own committente answer).
  7. P7 — Missing physical devices. The pushbutton, sonar and robot hardware do not exist yet: simulated counterparts must respect exactly the same message contracts, so that replacing them later does not touch the business logic.
  8. P8 — Race between timeout and detection. The window expiry and the detection event can occur almost simultaneously. The state machine semantics (the first transition taken wins, a late event finds no transition waiting and is dropped) must guarantee that the reservation is either discarded exactly once or the trip started exactly once, never both.

Test plans

The source code of this sprint is available in the repository: CargoService/Sprint1 (executable model: cargoservice_sprint1.qak).

Strategy. As promised in Sprint 0, the test plans of this sprint are automated: an external test client plays the role of the ioport boundary, using the same request/reply contract, and the hold is made observable and presettable (P5) so that every case has a machine-checkable Given and Then. Every plan asserts the exact expected reply and that exactly one reply is produced (guard for P1). The observation interactions are served when the service is quiescent, so the state assertions are placed at the quiescent points of each scenario.

TP1.1 — Load accepted on an empty hold (inherited from TP0.1)

Given the system started, the hold empty, state disengaged.
When the test sends one load_request.
Then the answer is load_accepted(SLOTID) with SLOTID in 1..4, and exactly one reply is produced.

TP1.2 — Load refused when the hold is full (deferred from Sprint 0)

Given the hold preset via preset_hold(full) (slots1-4 occupied), state disengaged.
When the test sends one load_request.
Then the answer is load_refused, exactly one reply, and get_hold shows the four slots occupied and the state still disengaged.

TP1.3 — Retry later when the system is engaged (deferred from Sprint 0)

Given an accepted request (system engaged), at least one further slot free.
When the test sends a second load_request.
Then the answer is load_retrylater(engaged), exactly one reply, and the first conversation is unaffected.

TP1.4 — Engagement timeout discards the reservation

Given an accepted request (system engaged) with no container ever placed in the sensor area.
When the prefixed 30 s window expires (the test tolerates the wait).
Then get_hold shows the state disengaged and all slots still free: the reservation has been discarded exactly once (P8).

TP1.5 — Complete load cycle

Given an accepted request on an empty hold.
When the sonar detects the container (detection event emitted).
Then the trip IOPort → slot5 (marking pause) → reserved slot → HOME completes, and at the end get_hold shows the reserved slot occupied and the state disengaged. In the integration variant the trip runs against the reused robot subsystem; in the unit variant the cargorobot is replaced by a mock that confirms each target, so the assertion isolates the business logic.

Note. The out-of-service branch has no test plan in this sprint: it is out of the declared goal and is deferred to Sprint 2 together with its logic.

Project (design)

Only now each element of the logical architecture is bound to a concrete technology, and every binding is justified by a problem identified in the Problem analysis.

Why qak (not assumed, justified)

The system is a set of distributed state machines that interact only by messages. Plain Java/Kotlin offers no first-class construct for actors, messages, events or timed transitions: encoding them by hand means threads, sockets and callbacks that bury the model. qak is an executable metamodel: QActors are Moore FSMs, messages are declared contracts (Request/Reply/Dispatch/Event), timed transitions exist natively (whenTime), and the same QActor can be deployed as a nano-service in one JVM or as a distributed component without changing the model. This is exactly the gap between our Problem analysis and runnable code, which is why the executable model is written in qak (full argument in Sprint 0, "Why qak").

The hold model in code

Coherently with the "Who knows the environment?" decision, the Hold POJO is used by cargoservice as a business registry: state of the hold, state of the slots, and the coordinates of the stations (exposed by getSlotRow/Col, getIoportRow/Col, getHomeRow/Col). The obstacle cells listed in its constructor are retained only as documentation of the layout: they are never used to navigate, since the navigation map is owned by RobotSmart26. The alignment between the two descriptions of the environment is a configuration-time duty, verified during the integration run.

Patterns

  • Facade / delega: cargoservice is the single entry point of the system; ioport, tests and (in future) the web GUI all talk to it only. The devices are delegated nano-services, which decouples the boundary from the business logic.
  • Adapter (Port/Adapter): cargorobot adapts the domain contract reachTarget to RobotSmart26's moverobot; the business logic depends on the port, not on the asset (SOLID/Clean Architecture).
  • Dependency inversion on passive resources: the business logic depends on IHold and IDisplay, not on the concrete classes.
  • Observer (events): the detection is a broadcast event, so the sonar does not know its consumers.

Deployment

Steps for an end user to run the Sprint 1 prototype:
  1. Clone the project repository: github.com/alessandromarcellini/CargoService;
  2. Start the robot environment (reused committente assets): in robotsmart26/yamls run docker compose -f unibobasic26.yaml up, then open http://localhost:8090/ in a browser to see the virtual robot scene;
  3. Start the robot service: in the robotsmart26 directory run ./gradlew run;
  4. Start the prototype contexts from CargoService/Sprint1, one process per context, with gradlew -b build2025.gradle run, setting application.mainClassName to the context main: it.unibo.ctxcargoservice.MainCtxcargoserviceKt, it.unibo.ctxdevices.MainCtxdevicesKt, it.unibo.ctxrobot.MainCtxrobotKt, and last it.unibo.ctxioport.MainCtxioportKt (the ioport is the manual demo driver);

Closing page

Architecture at the end of Sprint 1. The system is a set of message-based components on four contexts, plus the external navigation subsystem: the cargoservice (Facade over the whole system) owns the business logic and the hold model — slot states, hold state and station coordinates, made observable for the tests — while the map of the environment and the route planning live in the reused RobotSmart26 service; the ioport boundary and the JUnit client speak the load contract; the sonar mock produces the detection event; the cargorobot adapts the goal-level reachTarget contract to RobotSmart26's moverobot.

Links: executable model — sprint sources — decision log (RATIONALE.md).

Sprint 2 Goal

The objective of Sprint 2 is to evolve the prototype towards the real devices and the deferred requirements:
  1. implement the Out of service logic: sustained D > DFREE for at least 3 s, the display message, and the recovery behaviour (including stopping/resuming the robot), together with the 3 s persistence check of the detection measure;
  2. implement the LED blinking during the engagement window;
  3. move the sonar and LED logic towards the PicoW-like device (or its faithful simulation);
  4. realize the IOPort as a web GUI showing the hold state and the service messages;
  5. apply the committente answers to the open questions of this sprint (robot failure policy, return to HOME, engaged-state answer), adjusting the model where needed.
