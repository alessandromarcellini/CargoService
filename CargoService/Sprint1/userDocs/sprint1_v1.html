Introduction

Sprint 0 (available here) closed with the following results:
  • the requirements of the committente were copied and every ambiguity was turned into an explicit question to the committente (with the answers recorded);
  • the requirement analysis classified every entity of the domain (POJO, enum, actor, service), fixed the message contract of the load conversation, and selected RobotSmart26 as the robot asset to reuse, with an explicit comparison of the four assets offered by the committente;
  • a first executable model of the system was produced;
  • a first automated Test Plan (TP0.1, load accepted on an empty hold) was implemented and executed; two further cases (TP0.2 refused, TP0.3 retrylater) were specified but explicitly deferred to Sprint 1, because they require the state of the hold to be observable and presettable from a test;

The goal of this sprint, as declared in the closing page of Sprint 0, is reported below.
The goal of Sprint 1 is to create an executable prototype of the cargo service that implements the core business logic (loading and moving of the container), starting from the user load request up to the the container placement in the reserved slot, without worring about other small details like the Led blinking, the out of service possibility for the hold.

After finishing the sprint1 the system will be able to:
  1. Process load requests from the user, responding (with load_accepted, retry_later or load_refused) accordingly with the current state of the hold;
  2. Make the robot move across the hold to implement the container movement logic;
  3. Have a mock (simulation) of the devices of the ioport. The pushbutton will send messages to the system as the user would while the sonar will simulate the measuring of the distance in a proactive way, and the cargoservice will collect those informations during the 30s time windows;
  4. The sonar outofservice logic, the blinking of the led and the implementation of the ioport as a webgui are left for another sprint.

Requirements

The requirements are unchanged with respect to Sprint 0 and are reported verbatim.

The company asks us to build service named cargoservice that should work as follows.
The cargoservice is able to receive a request to load a container sent by some customer by using the pushbutton of the IOPort.

• It sends the answer retrylater, if the IOPort is currently occupied by a container or if the system is Out of service
• It rejects the request when the hold is already full, i.e. the slots1-4 are already occupied.
• Otherwise, it considers the system as engaged, detects a free slot and returns as answer the name of such a reserved slot. While engaged, the system must blink a Led.

When the request to load is accepted, the customer must move the container in the sensor area within prefixed amount of time (e.g. 30 secs), otherwise the systems becomes disengaged. Then, the cargoservice uses the cargorobot to move the container from the IOPort to the slot5 (for marking the container) and then to the reserved slot.

The service must also show on the display on the IOPort:
• the current state of the hold
• the message 'Service working', when it is all is going well
• the message 'Out of service' if the sonar sensor measures a distance D > DFREE for at least 3 secs (perhaps a failure of the sonar).

Requirement analysis

The requirement analysis was carried out in depth during Sprint 0 (document) and its results are assumed valid for this sprint. In particular, the following results are reused as they are:
  • the nature of the system: distributed and heterogeneous, deployed on three logical nodes (business logic with robot adapter, ioport, devices);
  • the entity model: IContainer/Container, ISlot/Slot, IDisplay/Display as passive objects; HoldState as an enumerative type; cargoservice as a service; ioport, sonar, cargorobot as actors;
  • the message contract of the load conversation (load_request with the three correlated answers) and the goal-level contract of the robot (reachTarget with its two correlated answers);
  • the committente clarifications already collected: the barcode of a container is 10 digits long; the dimensions of a container can be assumed integer; the functioning of the marker device can be emulated by a standard delay of 5 seconds in the container movement logic; the robot can be realized by reusing one of four assets provided by the committente (comparison in Sprint 0, choice: RobotSmart26).

Additional clarifications for Sprint 1 (questions to the committente)

The problem analysis of this sprint raised some points that the requirement text does not fix. Following the method, they are recorded as questions to the committente instead of being resolved silently. For each question we also state the interpretation we had adopted while the answer was pending, so that the impact of the committente's decision is clear.

  1. Robot failure policy. If the cargorobot cannot reach a target (answer targetUnreachable), what should the cargoservice do: retry the movement, abort the operation and free the reserved slot, or declare the system Out of service?
    Answer from the committente: if the robot cannot reach a target, the system must permanently stop accepting new load requests: the failure is a terminal condition, which we treat as entering the Out of service condition (permanent, as the answer on the recovery below fixes) — coherently with the requirement text, which associates retrylater with "the system is Out of service".
    Why we unify the robot-failure block with Out of service instead of inventing a new "blocked" state: (1) the requirement text already associates the retrylater answer with "the system is Out of service", so a system blocked by a robot failure keeps answering exactly as the published contract prescribes, with no new answer kind; (2) both conditions are permanent (no recovery, as the committente fixed), so they are behaviourally indistinguishable from the outside; (3) from the customer's viewpoint a service that can no longer move containers is out of service, and the display requirement (the 'Out of service' message) already covers how the condition must be communicated. In the model the whole policy is the target of a single transition (abort_transfer → out of service), which is what keeps a decision of this weight cheap to realize.

  2. Return to HOME. After depositing the container in the reserved slot, must the robot return to the HOME position, so that every load cycle starts from a known configuration?
    Answer from the committente: yes — once the container has been placed in the reserved slot, the robot returns to HOME when no other request is pending, so that a new cycle starts from a known configuration.
    Realization decision: the robot returns HOME after every cycle. Given the committente answer on the inhibition of the IOPort (inhibited while an operation is in progress) and the no-queueing policy, no customer request can be pending when the deposit completes: the committente condition is always satisfied, and a conditional skip-HOME branch could never trigger in production while leaving the robot parked at a slot — a worse starting configuration for the next cycle and for the demo.

  3. Second request while engaged. The requirement associates retrylater with "the IOPort is currently occupied by a container" and with the Out of service condition. It does not literally say what happens if a new request arrives while the system is engaged but the container has already been picked up from the IOPort.
    Adopted interpretation: the system serves one load operation at a time: any request received while the state is not disengaged is answered with retrylater, carrying the current state of the hold so that the display can explain the reason.
    Answer from the committente: new requests arriving while a load operation is in progress must be prevented: the cargoservice must inhibit the IOPort. Concretely: while the hold is engaged and the service is piloting the robot — hence it cannot stay listening for new conversations — the boundary itself answers retry later to every pushbutton press of the customer, without opening a conversation towards the service. The inhibition starts when the operation is accepted and ends when the system returns disengaged (see Problem analysis: inhibit_ioport/enable_ioport).
    Impact: the inhibition becomes the primary and, in this sprint, sufficient policy. We assume the boundary admits a single connected customer at a time (one user on the pushbutton, and later on the web GUI): since that single customer speaks only through the ioport and the ioport is inhibited for the whole engagement, no second load_request can reach the service along the normal path. The service-side retrylater guard is therefore not needed for production correctness here; we keep it for two reasons. It is required by the test phase — the automated TestPlans are clients that do not pass through the ioport (TP1.3 reaches the engaged service directly), and the guard gives them the well-defined answer instead of a silently deferred request; and it is the extension hook for a possible future in which the web GUI admits several concurrent users, where a second request could genuinely reach the busy service. Two clarifications complete the picture: the system does not queue load requests — a request that reaches the busy service receives an immediate retrylater, never a deferred acceptance — and the ioport discriminates the reason of a retry later through the argument of load_retrylater(HOLDSTATE) (engaged vs out_of_service), which it already uses for its display messages: no additional reason-carrying interaction is needed. Again, the change costs little because it is a new declared interaction in the model, not a rework of the components.

  4. Restore from Out of Service The requirements do not talk about in which conditions the system can be restored from the Out of service state.
    Answer from the committente: there is no recovery: once the hold is Out of service, the service remains in that state without changing it (bringing the system back is a maintenance intervention, outside the scope of the service). Together with the robot-failure answer, Out of service is therefore a terminal condition reachable both from the sonar failure (deferred to Sprint 2) and from a robot failure.
    Test note: the no-recovery constraint concerns the production behaviour; for test purposes the state can be reset through the test-support preset_hold interaction (pure controllability infrastructure of the TestPlans).

  5. Sonar Behavior Does the sonar needs to wait the cargoservice's message to start detecting the distance of the container?
    Answer from the committente: no — the sonar must be a proactive entity, decoupled from the rest of the application: it publishes on its own initiative the distance it measures as events, and the cargoservice subscribes to those events only once it has taken charge of a request, for the whole 30 s time window. No trigger message exists.
    A further committente point: until the physical sonar on the PicoW-like device is introduced, the emitted distances must not be random — the TestPlans need deterministic, reproducible situations — but produced from a statically configurable source (the binding is a Design concern). Impact: the checkMeasurement dispatch disappears from the contract and for now the detection interpretation moves into the service: D < DFREE/2 sustained for a reasonable time (~3 s), realized as three consecutive below-threshold measurements at the 1 Hz publication rate (see Problem analysis). Both changes touch only declared interactions and one component behaviour in the executable model: this is exactly the flexibility that working on a model buys us.

Problem analysis

This section defines the logical architecture of the Sprint 1 slice of the system. Following the method, it deals only with abstract concepts (actors, messages, events, services, contexts): every binding to a concrete technology is deferred to the Project section, where each choice is justified by a problem identified here.

Scope of the analysis

In scope for this sprint:
  • the load conversation (request and the three answers) with mutually exclusive branches;
  • the engagement lifecycle, including the prefixed waiting window (30 s) after which the system becomes disengaged;
  • the detection of the container in the sensor area (simulated measurement);
  • the movement of the container: IOPort → slot5 (marking pause) → reserved slot → HOME, and the bookkeeping of the hold;
  • the observability of the hold state, needed to automate the test plans deferred from Sprint 0.
Out of scope, deferred to Sprint 2: the sonar-driven Out of service detection (sustained D > DFREE for 3 s — with no recovery, as the committente fixed), the LED blinking, the physical devices, and the IOPort as a web GUI.

Who knows the environment?

Before fixing the structure, one question must be answered explicitly, because it decides what the business model contains: who owns the knowledge of the physical environment? Two different kinds of spatial knowledge coexist in this system:
  • navigation knowledge: which cells are free or blocked by obstacles, and how to compute a route between two points. This knowledge is needed only to move;
  • business knowledge: which slots exist, whether each one is free or occupied, the state of the hold, and the coordinates of the relevant stations (IOPort, slot1..5, HOME). This knowledge is needed to answer requests and to command the trip.
Decision: the navigation knowledge (the map of the environment) belongs to RobotSmart26, which needs it to plan and execute movements; the business model owned by cargoservice keeps only the state of the slots, the state of the hold and the coordinates of the stations.
Why: single responsibility — the business logic never plans a route, so it has no reason to own a map; duplicating the map in two places would create a consistency problem between the business copy and the environment the robot actually navigates. The residual risk (the station coordinates kept by the business model must agree with the robot subsystem's environment) is a configuration-time alignment, accepted and logged.

The hold model backtracking

The decision above made us backtrack on how the hold information is modelled, with respect to Sprint 0. There, the hold was a single passive entity (an IHold interface with a Hold object) holding both the state of the slots and a cell map of the environment. In this sprint that single entity is dissolved into two knowledges with two owners:
  • the business registry, owned by cargoservice: state of the hold, state of the slots (slot1..4, slot5), and the coordinates of the stations (IOPort, slot5, slot1..4, HOME) — everything the service needs to answer requests and to command the trip;
  • the logical map of the environment, owned by the navigation subsystem made available by the committente: cells, obstacles and routes — everything needed to move, and nothing the business logic ever reads.
The alignment between the two descriptions (the station coordinates in the registry must denote the same places of the map the robot navigates) is performed out of band, at initial system configuration time: both are generated from the same hold layout when the system is installed, and no runtime protocol keeps them synchronized. This is a deliberate choice, justified twice: by simplicity (Occam's razor — no synchronization interaction, no consistency protocol) and by the acceleration of the development (no new component to build, the reused navigation subsystem stays untouched). The alignment is verified in the integration run (see Test plans).

Reversibility. This split is not a one-way door. Two future needs could justify re-unifying the hold into a single entity that owns both the state and the map: a business logic that must reason about space (choose the nearest free slot, forbid areas of the hold, order deposits by distance), or the appearance of more writers than cargoservice (say, an unloading service updating the same slots). If either arrives, the unified Hold can be reintroduced at a contained cost, and the reason is architectural, not accidental: the hold state registry lives behind the cargoservice (no other component reads it directly), and every interaction of the system names stations ("the reserved slot", "HOME"), never the data structures that store them. Re-unifying therefore changes who owns a datum and how the service resolves a name into a position — while every declared message of the system stays exactly as it is. This is one more dividend of working on a model: responsibilities can be redistributed without renegotiating the contracts.

Where does the business logic live?

The method requires this question to be answered explicitly. No pre-existing entity can host the business logic: the ioport is a boundary, the sonar is a detector, the cargorobot executes one goal at a time, the external robot subsystem only navigates. The business logic therefore lives in cargoservice, which owns:
  • the answer policy: accepted / refused / retrylater, decided against the hold model;
  • the engagement lifecycle: the transition disengaged → engaged on acceptance, and back on completion or timeout;
  • the trip plan: the sequence IOPort → slot5 (marking pause) → reserved slot → HOME, as the requirement sentence "the cargoservice uses the cargorobot to move the container..." dictates. Since the business model owns the station coordinates, cargoservice also resolves each named station into the position sent to the robot;
  • the hold state bookkeeping: reservation (kept in the service memory while engaged), occupation on deposit, discard of the reservation on timeout or failure.
The hold model is deliberately a passive object with a single owner (cargoservice): a single writer avoids consistency problems between concurrent requests, and a single place must be read to know the state of the system — which is exactly what the automatable test plans need.

How is the hold state observed from the outside?

The hold state lives inside cargoservice. Two ways exist to make it visible to an external observer (a test, and in perspective the display):
  • push — the service publishes a hold-state event at every change: observers are fully decoupled, but assertions can fire at arbitrary instants and every state change pays the publication;
  • pull (chosen) — the state is returned on request (get_hold), and the request is served only at the quiescent points of the service. A test therefore reads the state exactly where an assertion is meaningful, and the pending reply doubles as a synchronization barrier (the engagement-timeout plan literally waits on it).
We choose pull, for three reasons. First, it matches the consumer we actually have in this sprint: a test asserts at precise points of a scenario, so an on-request observation delivers the state exactly when it is meaningful, while a stream of push events would force the test to filter and correlate them. Second, it protects the single writer: the service exposes its state only at quiescent points, so an observation can never interleave with a conversation half-way through and photograph an inconsistent hold. Third, it costs nothing beyond one request/reply pair — no publication at every change, no subscriber bookkeeping. The push alternative is not discarded forever: the display requirement ("show the current state of the hold") will naturally push the hold state towards the IOPort in Sprint 2, when a consumer that wants continuous updates actually exists — and that evolution stays cheap while the decision lives in the model.

What is in and what is deferred about Out of service. The Out of service condition is part of this sprint: a robot failure marks the hold OUT_OF_SERVICE, and since nothing in production ever resets that mark, from that moment every load request receives the answer retrylater(out_of_service) — exactly the behaviour the requirement text prescribes. What this sprint does not yet realize is the second way of entering the condition, the sonar failure: detecting D > DFREE sustained for 3 s (the event outOfService(none) is declared as its future carrier, but no component emits it yet) and showing the 'Out of service' evidence on the display. Both arrive with Sprint 2; the detection mechanism is already specified, since it mirrors the persistence counter used for the container detection.

The load conversation (abstract protocol). The ioport sends load_request. The cargoservice answers with exactly one of the three replies. On acceptance the system becomes engaged, inhibits the ioport (while inhibited, the boundary answers retry later to the customer by itself, since the service is busy piloting the robot and cannot serve new conversations) and opens the waiting window, during which it listens to the distance events published by the sonar: the first event with D < DFREE/2 closes the window and starts the trip; the window expiring closes it in the other direction, discarding the reservation and re-enabling the ioport. The trip is a sequence of reachTarget requests, each confirmed before the next: IOPort (pick up), slot5, a marking pause (~5 s, per committente clarification), the reserved slot, then HOME (per the committente answer on the return to HOME — its 'no pending request' condition is always true, given the ioport inhibition and the no-queueing policy). After the deposit, the slot is marked occupied, the ioport is re-enabled and the system returns disengaged. If a trip leg fails, the hold is marked Out of service and the ioport is re-enabled: from then on every request is answered retrylater(out_of_service), forever (no recovery exists in production).

Test plans

The source code of this sprint is available in the repository: CargoService/Sprint1 (multi-context model: cargoservice_sprint1.qaktt; the single-context executable prototype that collapses every context into one for the demo and the unit tests: cargoservice_sprint1_prototype.qak).

Strategy. As promised in Sprint 0, the test plans of this sprint are automated: an external test client plays the role of the ioport boundary, using the same request/reply contract, and the hold is made observable and presettable so that every case has a machine-checkable Given and Then. Every plan asserts the exact expected reply and that exactly one reply is produced (guard for P1). The observation interactions are served when the service is quiescent, so the state assertions are placed at the quiescent points of each scenario.

How the observability of the system is achieved (new interactions)

The automatable-test requirement implies an observable and controllable system. Following the pull decision taken in the Problem analysis ("How is the hold state observed from the outside?"), two interactions were added to the contract of the service — they do not touch the business conversation and are served only in the waiting state:
  • get_hold(none) → hold_state(HOLDSTATE) (request/reply): returns the observable description of the hold, e.g. disengaged_free_occupied_free_free (state + the four slots). Because a request arriving while the service is busy is deferred until quiescence, the pending reply is also a synchronization barrier: a test that asks for the hold state receives the answer exactly at the point where the assertion is meaningful (the engagement-timeout plan waits on it for the whole window);
  • preset_hold(full|empty) → preset_done(done) (request/reply): builds the Given of every plan (full hold for the refuse case, empty hold elsewhere), making each TestPlan independent from the previous ones. It is also the only way to clear the out-of-service mark of the hold: a test-only controllability backdoor, explicitly excluded from the production behaviour (no recovery exists in production).
The third leg of controllability is the sonar remodelling: the detection reaches the service as broadcast events, so a test can publish the very same distance events the real sonar would publish (three consecutive below-threshold values, since the detection must be sustained) — no test-only hook enters the service.

When the test sends a second load_request — note that the test client does not pass through the (now inhibited) ioport: it is precisely the kind of non-ioport client for which the service-side no-queueing guard is kept (see the Problem analysis and the design notes), and this plan is what makes that guard observable.
Then the answer is load_retrylater(engaged), exactly one reply, and the first conversation is unaffected.

TP1.4 — Engagement timeout discards the reservation

Given an accepted request (system engaged) with no container ever placed in the sensor area: deterministically, no distance event below DFREE/2 is published during the window (in the unit configuration no sonar is deployed at all; in a prototype run the same situation is a configuration of the static distance table).
When the prefixed 30 s window expires (the test tolerates the wait).
Then get_hold shows the state disengaged and all slots still free: the reservation has been discarded exactly once.

TP1.5 — Complete load cycle

Given an accepted request on an empty hold.
When the sonar publishes a below-threshold distance (D < DFREE/2) sustained for three consecutive measurements (~3 s): the detection, per the remodelled contract.
Then the trip IOPort → slot5 (marking pause) → reserved slot → HOME completes, and at the end get_hold shows the reserved slot occupied and the state disengaged. In the integration variant the trip runs against the reused robot subsystem; in the unit variant the cargorobot is replaced by a mock that confirms each target, so the assertion isolates the business logic.

TP1.6 — Integration: cargorobot with the reused navigation subsystem

The plans above isolate the business logic; the interaction between the cargorobot adapter and the reused navigation subsystem (robotsmart26) must be verified too — it is where the out-of-band coordinate alignment of the hold model backtracking can actually break.
Given the robot environment and the navigation service running (Deployment, steps 2-3), the hold empty, the virtual robot at HOME.
When an accepted load cycle runs (detection event published as in TP1.5).
Then every reachTarget leg is confirmed by a moverobotdone of the navigation subsystem (the virtual robot visibly performs IOPort → slot5 → reserved slot → HOME), and at the end get_hold shows the reserved slot occupied and the state disengaged.
Implementation: test/it/unibo/test/IntegrationTestPlanSprint1.kt, over the dedicated deployment cargoserviceintegrationtest.pl (the REAL cargorobot adapter, the external robotsmart context, MockIoport for the boundary commands, detection injected as in TP1.5). It is excluded from the default gradlew test and launched on demand with gradlew integrationTest, with the robot environment up (Deployment, steps 2-3); a fail-fast check explains the prerequisites if robotsmart26 is down. Note the diagnostic value of the failure mode: a coordinate misalignment makes a trip leg end targetUnreachable, so the service lands in the terminal out-of-service state and the plan reports hold_state(out_of_service_...) — the exact defect this plan exists to catch.

JUnit implementation (TestPlanSprint1)

The five plans are implemented as one JUnit 4 class, test/it/unibo/test/TestPlanSprint1.kt, runnable with a single command (gradlew test) and with no external process: the test JVM starts, once, the qak test configuration described by cargoservicetest.pl.

The test configuration (unit variant of TP1.5). The configuration deploys the real cargoservice together with two test doubles, all co-located in ctxcargoservice:
  • MockCargorobot replaces the robot adapter: it honours the same reachTarget/targetReached contract but confirms every target, so the assertions isolate the business logic (no robotsmart subsystem needed);
  • MockIoport absorbs the inhibit_ioport/enable_ioport boundary commands the service sends around every operation. No sonar is deployed: each plan injects the very sonar_distance(D) events the proactive sonar would publish (three consecutive below-threshold values for a detection), which makes every plan deterministic — in particular TP1.4, which must observe no detection for 30 s.
Because a qak actor is bound to its context only by the deployment description (location transparency), the service code under test is byte-for-byte the production one: only the .pl file changes. The test class plays the ioport boundary over TCP (port 8030) using the same request/reply contract of the model.

Changes introduced to make the TestPlans executable

  1. Mutually exclusive answer branches. The defect inherited from Sprint 0 (sequential checks in handle_load_request, so one request could produce two replies) is corrected: the three branches are now an if / else-if / else chain, and every branch ends the conversation with exactly one reply.
  2. Observability/controllability messages. New in the executable model: get_hold(none) with reply hold_state(HOLDSTATE) (e.g. disengaged_free_occupied_free_free) and preset_hold(full|empty) with reply preset_done(done), served by the states serve_get_hold/serve_preset_hold reachable from waiting only. They give every plan a machine-checkable Given and Then without touching the business conversation.
  3. Lowercase payload atoms (message-as-contract). The retrylater answer now carries load_retrylater(engaged) instead of load_retrylater(ENGAGED): an uppercase argument is parsed as a logical variable in the message term, so it is not a value a test (or the ioport) can match on. The ioport display logic was updated accordingly.
  4. Timer bug found by TP1.4. The first realization measured the window with the qak state timer (whenTime). TP1.4 revealed it never fired: the qak TimerActor resolves its owner actor from the timeout-event name with substringBeforeLast("_"), so with a state named accept_load_request the generated event local_tout_cargoservice_accept_load_request is parsed as owner "cargoservice_accept_load", which does not exist — the timer thread loops forever and the 30 s window never expires. Rather than work around the parser, we dropped the state timer and armed a one-shot timer once, at the acceptance (a single background coroutine that sleeps 30 s and then dispatches cargoservice_window_expired; see the realization notes in the Project section). TP1.4 now measures the window at ~30.0 s. This is a concrete return on the "test plans must be automatable" requirement: the defect was invisible in any manual run.

How to run the TestPlans (step-by-step instructions for the committente)

The following procedure runs the automated TestPlans in their unit variant: it requires no Docker, no robot environment and no other process — the test JVM starts everything it needs by itself.

  1. Open a terminal in the sprint folder: CargoService/Sprint1 (the folder that contains build.gradle, cargoservicetest.pl and sysRules.pl). The tests must be launched from here: the test configuration reads the two .pl files from the working directory.
  2. Make sure port 8030 is free. If a cargoservice context (or a previous, interrupted test run) is still alive, close it first: the test JVM must be able to open TCP port 8030 itself.
  3. Launch the tests:
    • Windows: gradlew.bat -b build.gradle test
    • Linux / macOS: ./gradlew -b build.gradle test
    The first launch may take a few minutes because Gradle downloads the declared dependencies; the following launches are fast.
  4. Wait for the run to finish — do not interrupt it. The whole suite lasts about one minute: TP1.4 intentionally waits the full 30 s engagement window (that wait is the test). The console shows the progress of each plan (passed/failed) together with the messages exchanged with the service.
  5. Read the outcome. Expected result: BUILD SUCCESSFUL with 5 tests completed, 0 failed. A browsable report is generated in build/reports/tests/test/index.html, with the standard output of every plan attached.
  6. Integration variant (TP1.6, optional). To verify the interaction with the real navigation subsystem, first start the robot environment (Deployment, steps 2-3: Docker + robotsmart26), then launch gradlew integrationTest: the plan runs the complete load cycle against robotsmart26 (the virtual robot moves on http://localhost:8090) and verifies the coordinate alignment of the hold model backtracking. The default gradlew test remains deliberately independent from that setup.

class TestPlanSprint1 {

    companion object {
        private const val HOST = "localhost"
        private const val PORT = 8030          // ctxcargoservice, from cargoservice_sprint1.qak

        @BeforeClass
        @JvmStatic
        fun startTestConfiguration() {
            CommUtils.outcyan("TestPlanSprint1 | starting the qak test configuration (cargoservicetest.pl)")
            thread(isDaemon = true) {
                runBlocking {
                    QakContext.createContexts(
                        HOST, this, "cargoservicetest.pl", "sysRules.pl", "ctxcargoservice"
                    )
                }
            }
            Thread.sleep(5000)  // let the context server and the actors boot
        }
    }

    private var conn: Interaction? = null

    @Before
    fun setup() {
        try {
            conn = TcpClientSupport.connect(HOST, PORT, 10)
        } catch (e: Exception) {
            fail("TCP connection error towards the test configuration ($HOST:$PORT): ${e.message}")
        }
    }

    @After
    fun teardown() {
        conn?.close()
    }

    // ---------------------------------------------------------------- helpers

    private fun request(msgId: String, content: String): String {
        val envelope = "msg($msgId,request,testunit,cargoservice,$content,1)"
        val reply = conn!!.request(envelope)
        assertNotNull("no reply received for $msgId", reply)
        return reply.toString()
    }

    private fun loadRequest(): String = request("load_request", "load_request(none)")

    /** Presets the hold (Given); the reply is also a quiescence barrier. */
    private fun presetHold(config: String) {
        val reply = request("preset_hold", "preset_hold($config)")
        assertTrue("preset_hold($config) not confirmed (spurious reply pending?): $reply",
                   reply.contains("preset_done"))
    }

    /** Reads the observable hold description (Then + quiescence barrier). */
    private fun getHoldDescription(): String {
        val reply = request("get_hold", "get_hold(none)")
        assertTrue("expected hold_state but got (spurious reply pending?): $reply",
                   reply.contains("hold_state"))
        val m = Regex("""hold_state\(\s*([a-z_]+)\s*\)""").find(reply)
        assertNotNull("could not parse the hold description from: $reply", m)
        return m!!.groupValues[1]       // e.g. disengaged_free_occupied_free_free
    }

    /** Publishes the same distance events the proactive sonar would: the
        detection requires D < DFREE/2 SUSTAINED for three consecutive
        measurements (~3 s at the 1 Hz rate). */
    private fun injectContainerDetected() {
        repeat(3) {
            conn!!.forward("msg(sonar_distance,event,sonar,none,sonar_distance(1),1)")
            Thread.sleep(300)
        }
    }

    private fun slotIdOf(reply: String): Int? =
        Regex("""load_accepted\(\s*(\d+)\s*\)""").find(reply)?.groupValues?.get(1)?.toIntOrNull()

    private fun occupiedCount(holdDesc: String): Int =
        Regex("occupied").findAll(holdDesc).count()

    // -------------------------------------------------------------- TestPlans

    /** TP1.1 — load accepted on an empty hold (inherited from TP0.1). */
    @Test(timeout = 60000)
    fun test1_loadAcceptedOnEmptyHold() {
        presetHold("empty")

        val reply = loadRequest()
        assertTrue("expected load_accepted but got: $reply", reply.contains("load_accepted"))
        assertFalse("wrong branch taken: $reply",
                    reply.contains("load_refused") || reply.contains("load_retrylater"))
        val slotId = slotIdOf(reply)
        assertNotNull("could not read the reserved slot id from: $reply", slotId)
        assertTrue("reserved slot id out of range (expected 1..4): $slotId", slotId!! in 1..4)

        // close the engagement (detection + full trip) so that the system is
        // quiescent again; getHoldDescription() is also the one-reply guard
        injectContainerDetected()
        val hold = getHoldDescription()
        assertTrue("system not disengaged at the end of the cycle: $hold", hold.startsWith("disengaged"))
    }

    /** TP1.2 — load refused when the hold is full (deferred from Sprint 0). */
    @Test(timeout = 60000)
    fun test2_loadRefusedOnFullHold() {
        presetHold("full")

        val reply = loadRequest()
        assertTrue("expected load_refused but got: $reply", reply.contains("load_refused"))
        assertFalse("wrong branch taken: $reply",
                    reply.contains("load_accepted") || reply.contains("load_retrylater"))

        val hold = getHoldDescription()
        assertTrue("expected disengaged: $hold", hold.startsWith("disengaged"))
        assertEquals("the four slots must still be occupied: $hold", 4, occupiedCount(hold))
    }

    /** TP1.3 — retry later when the system is engaged (deferred from Sprint 0). */
    @Test(timeout = 90000)
    fun test3_retryLaterWhenEngaged() {
        presetHold("empty")

        val first = loadRequest()
        assertTrue("first request must be accepted: $first", first.contains("load_accepted"))
        Thread.sleep(500)   // let the service enter the engagement window

        val second = loadRequest()
        assertTrue("expected load_retrylater but got: $second", second.contains("load_retrylater"))
        assertTrue("the retrylater answer must carry the engaged state: $second",
                   second.contains("engaged"))
        assertFalse("wrong branch taken: $second",
                    second.contains("load_accepted") || second.contains("load_refused"))

        // the first conversation completes normally
        injectContainerDetected()
        val hold = getHoldDescription()
        assertTrue("expected disengaged: $hold", hold.startsWith("disengaged"))
        assertEquals("only the slot of the FIRST conversation must be occupied: $hold",
                     1, occupiedCount(hold))
    }

    /** TP1.4 — engagement timeout discards the reservation. */
    @Test(timeout = 90000)
    fun test4_engagementTimeoutDiscardsReservation() {
        presetHold("empty")

        val reply = loadRequest()
        assertTrue("request must be accepted: $reply", reply.contains("load_accepted"))

        val start = System.currentTimeMillis()
        val hold = getHoldDescription()   // served only after the 30 s window
        val elapsed = System.currentTimeMillis() - start

        assertTrue("expected disengaged after the timeout: $hold", hold.startsWith("disengaged"))
        assertEquals("the reservation must be discarded (no slot occupied): $hold",
                     0, occupiedCount(hold))
        assertTrue("the window closed too early (${elapsed} ms)", elapsed >= 25000)
    }

    /** TP1.5 — complete load cycle (unit variant). */
    @Test(timeout = 90000)
    fun test5_completeLoadCycle() {
        presetHold("empty")

        val reply = loadRequest()
        assertTrue("request must be accepted: $reply", reply.contains("load_accepted"))
        val slotId = slotIdOf(reply)
        assertNotNull("could not read the reserved slot id from: $reply", slotId)

        injectContainerDetected()
        val hold = getHoldDescription()   // deferred until the trip is over

        assertTrue("expected disengaged at the end of the cycle: $hold", hold.startsWith("disengaged"))
        val parts = hold.split("_")       // [disengaged, slot1, slot2, slot3, slot4]
        assertEquals("unexpected hold description: $hold", 5, parts.size)
        assertEquals("the RESERVED slot ($slotId) must be occupied: $hold",
                     "occupied", parts[slotId!!])
        assertEquals("exactly one slot must be occupied: $hold", 1, occupiedCount(hold))
    }

Project (design)

Only now each element of the logical architecture is bound to a concrete technology, and every binding is justified by a problem identified in the Problem analysis.

Why qak (not assumed, justified)

The system is a set of distributed state machines that interact only by messages. Plain Java/Kotlin offers no first-class construct for actors, messages, events or timed transitions: encoding them by hand means threads, sockets and callbacks that bury the model. qak is an executable metamodel: QActors are Moore FSMs, messages are declared contracts (Request/Reply/Dispatch/Event), timed transitions exist natively (whenTime), and the same QActor can be deployed as a nano-service in one JVM or as a distributed component without changing the model. This is exactly the gap between our Problem analysis and runnable code, which is why the executable model is written in qak (full argument in Sprint 0, "Why qak").

Two gaps between the FSM model and qak, and how we close them

The behaviour was designed as a Moore machine; turning it into qak code raises two concrete questions — how to branch on a value and how to measure a fixed duration — because qak does not express either directly. Each is closed with a small, deliberate idiom.
  • Branching on a value, not on a message kind (the detection). A qak transition fires on the kind of a message (its name), never on the values it carries. The detection rule — "the distance is below threshold" — is a test on a value, so it cannot be a transition guard. We realize it as an evaluation state, eval_distance: every sonar_distance event routes the FSM into this state, which reads D and updates a persistence counter. Three consecutive below-threshold readings (~3 s at the 1 Hz publication rate, the requirement's "reasonable time") start the trip; a far reading resets the counter; every other case routes back to the engagement window. The counter, not the transition, carries the decision.
  • Measuring a fixed 30 s from a single instant (the window). qak's native timer is the state timer (whenTime): it is (re)armed every time the state is entered and discarded on exit. That is the wrong tool here, because the engagement window is not spent sitting still — each sonar measurement makes the FSM leave the window for eval_distance and come back, which would restart a state timer and stretch the 30 s indefinitely. We therefore arm a one-shot timer once, at the acceptance: a single background coroutine sleeps 30 s and then sends the dispatch cargoservice_window_expired, which the engaged states receive and act on by switching to disengaged. Because the evaluation round-trips re-enter the window with a plain Goto (they never re-arm the timer), the 30 s are counted from the acceptance instant regardless of how many measurements or guarded requests arrive in between. This is sufficient here since a single customer is served at a time, so engagements never overlap.

Contexts and node allocation

Two decisions shape the deployment. First, ctxcargoservice was on port 8020, which collides with the context of the reused robotsmart service (fixed by the asset at 8020): our contexts are moved to the 8030 range; the test configuration is updated accordingly. Second, after the committente call, cargoservice and cargorobot execute on the same node and in the same context: the committente wants to spare resources in this first deployment (one JVM and one context). The choice costs nothing architecturally: the two components interact by declared messages in both arrangements, so the committente can later decide to separate them onto different nodes by changing only this deployment description.

The hold model backtracking — technology binding

The backtracking itself (business registry in cargoservice, logical map in the navigation subsystem, out-of-band alignment at configuration time) is a logical decision and is presented in the Problem analysis. Here only its binding: the navigation subsystem is realized by the reused asset RobotSmart26, whose internal sub-service maintains the map model of the environment the robot operates in; the IHold interface and the Hold POJO of Sprint 0 are removed from the code, and the business registry lives in the cargoservice actor (slot list, hold state, station coordinates). The configuration-time alignment between the registry and the RobotSmart26 map is verified by the integration TestPlan (TP1.6).

Device integration: the physical devices speak MQTT, without per-device adapters

The devices installed on the PicoW-like board (sonar and led) emit events and communicate with the qak application component cargoservice through the MQTT protocol (pub/sub). Two realizations were considered:
  • per-device adapters — one component per device translating the messages published by the board into the messages of the model. Discarded: the qak metamodel intrinsically manages MQTT message exchange, so the adapters would duplicate a capability the runtime already offers, adding components (and failure points) without adding decoupling;
  • direct integration (chosen) — the device events enter the system as the declared events of the model (e.g. sonar_distance(D)), carried by MQTT: the message-as-contract discipline is preserved, and the physical sonar will replace the simulated one without touching any consumer.
The only argument in favour of the adapters would come from the clean-architecture principles (keeping the transport concern entirely outside the application boundary); we record it, and the model keeps the decision cheap to revisit if that purity ever becomes a requirement.

Status in this sprint. To be precise about what runs today: no MQTT flow exists yet, because the physical devices are outside the sprint scope. The simulated sonar is realized as a qak actor in ctxdevices, and its sonar_distance events travel on the ordinary qak event channel across contexts; no broker appears in the Deployment. What the decision above binds is the future: when the pico devices arrive (Sprint 2), their MQTT-published events will enter the system as the same declared events — no consumer changes, no adapter appears — because the MQTT capability is native in the qak runtime. This is also why the substitution will be cheap: contract unchanged, only the carrier changes.

Sonar distance source (deterministic prototype runs). Until the physical sonar is introduced, the distances published by the simulated sonar are not random: they are read from a statically configurable table (a simple configuration file listing the distances to emit, in order), so that every prototype/demo execution is deterministic and reproducible, as the committente requires. The table governs the prototype and demo runs (and the integration plan TP1.6, which exercises the real chain); the unit TestPlans do not use the simulated sonar at all: there the mock publishes nothing and each plan injects the distance event it needs. When the pico sonar arrives, it will publish really-measured distances on the same contract — the table disappears, nothing else changes.

Patterns

  • Facade / delega: cargoservice is the single entry point of the system; ioport, tests and (in future) the web GUI all talk to it only. The devices are delegated nano-services, which decouples the boundary from the business logic.
  • Adapter (Port/Adapter): cargorobot adapts the domain contract reachTarget to RobotSmart26's moverobot; the business logic depends on the port, not on the asset (SOLID/Clean Architecture).
  • Dependency inversion on passive resources: the business logic depends on IDisplay, ISlot and IContainer, not on the concrete classes.
  • Observer (events): the detection is a broadcast event, so the sonar does not know its consumers.
